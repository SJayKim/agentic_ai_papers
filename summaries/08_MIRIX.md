# MIRIX: Multi-Agent Memory System for LLM-Based Agents

> **논문 정보**: Yu Wang, Xi Chen (MIRIX AI)
> **arXiv**: 2507.07957v1 (2025.07.10) | **학회**: -
> **코드**: https://mirix.io/

---

## Problem

현재 대부분의 LLM 기반 개인 비서는 현재 프롬프트 윈도우를 넘어서는 상태를 유지하지 못하며, 맥락이 명시적으로 재공급되지 않는 한 지속적인 메모리를 보유하지 않는다. 이는 사용자가 시간이 지남에 따라 진화하고 개인화되는 비서를 기대하는 실세계 환경에서 장기적 사용성을 심각하게 저해한다. 기존 메모리 보강 시스템들은 세 가지 근본적 한계를 가진다: (1) 구성적 메모리 구조의 부재 — 대부분이 단일 평탄(flat) 저장소에 모든 이력 데이터를 저장하여 전문화된 메모리 유형(절차적, 에피소드적, 의미적)으로의 라우팅이 없어 검색이 비효율적이다. (2) 멀티모달 지원 부족 — 텍스트 중심 메모리 메커니즘은 이미지, 인터페이스 레이아웃, 지도 등 비언어적 입력을 처리하지 못한다. (3) 확장성과 추상화의 결여 — 원시 입력(특히 이미지)을 그대로 저장하면 금지적 수준의 메모리 요구량이 발생하며, 핵심 정보만 요약하여 보존하는 추상화 계층이 없다.

---

## Motivation

인간의 인지는 대화 회상, 패턴 인식, 과거 경험 기반 행동 적응 등 메모리에 크게 의존한다. LLM 에이전트에서도 일관되고 개인화된 상호작용, 피드백 학습, 반복 질의 회피를 위해 메모리 메커니즘이 필수적이다. 기존 접근법 중 지식 그래프 기반 시스템(Zep, Cognee)은 구조적 관계 표현에 적합하나 순차적 사건, 감정 상태, 멀티모달 입력 모델링에 실패한다. 평탄 메모리 아키텍처(Letta, Mem0, ChatGPT Memory)는 벡터 DB를 활용한 텍스트 청크 저장/검색에 그쳐 전문화된 메모리 유형 간 라우팅이 불가능하다. 저자들은 인지과학의 에피소드, 의미, 절차적 메모리 구분에서 영감을 받아, 효과적인 라우팅(Routing)과 검색(Retrieving)이 메모리 보강 에이전트의 핵심 역량이라 주장하며, 6종의 전문화된 메모리 컴포넌트와 멀티에이전트 조율 구조를 설계한다.

## Method

1. **6종 메모리 컴포넌트 설계**: Core Memory(항상 에이전트에 노출되는 고우선 지속 정보 — persona/human 블록), Episodic Memory(타임스탬프 기반 사건/경험 로그), Semantic Memory(시간 독립적 추상 지식/엔티티), Procedural Memory(목표 지향 절차·워크플로·스크립트), Resource Memory(문서/전사본/멀티모달 파일), Knowledge Vault(자격증명·주소·연락처 등 민감 정보의 보안 저장소).

2. **계층적 내부 구조**: 각 메모리 컴포넌트는 계층 구조로 조직된다. 예를 들어 Episodic Memory는 summary/details 필드를, Semantic Memory는 name/description으로 구성된다.

3. **Multi-Agent Architecture**: 6개의 Memory Manager(각 메모리 컴포넌트 전담)와 1개의 Meta Memory Manager(태스크 라우팅 담당), 1개의 Chat Agent(대화 인터페이스)로 총 8개 에이전트가 협업한다.

4. **Memory Update Workflow**: 사용자 입력 수신 → 메모리 베이스 자동 검색 → 검색 결과 + 입력을 Meta Memory Manager에 전달 → 관련 Memory Manager들에 라우팅 → 각 Memory Manager가 병렬로 해당 메모리 업데이트(중복 방지) → Meta Memory Manager에 완료 보고.

5. **Active Retrieval 메커니즘**: 명시적 메모리 검색 트리거 없이도 자동 작동. (1단계) 에이전트가 입력 컨텍스트에서 현재 토픽을 생성, (2단계) 해당 토픽으로 6개 메모리 컴포넌트 각각에서 top-10 관련 항목 검색, 검색 결과를 소스 태그와 함께 시스템 프롬프트에 주입.

6. **다중 검색 함수 지원**: embedding_match, bm25_match, string_match 등 다양한 검색 전략을 제공하며, 에이전트가 맥락에 따라 최적 검색 방법을 선택한다.

7. **멀티모달 처리**: 스크린샷 입력 시 1.5초 간격 캡처, 시각적 유사도 기반 중복 제거, 20장의 고유 스크린샷 수집 시 메모리 업데이트 트리거. 스트리밍 업로드 전략으로 지연 시간 최소화.

8. **Core Memory 용량 관리**: 메모리 크기가 용량의 90%에 도달하면 제어된 재작성(rewrite) 프로세스를 트리거하여 핵심 정보 손실 없이 컴팩트함을 유지.

## Key Contribution

1. **포괄적 모듈형 메모리 시스템**: 기존 메모리 아키텍처의 한계를 분석하고, 6종 전문화된 메모리 컴포넌트 + 8개 에이전트로 구성된 최초의 포괄적 모듈형 메모리 시스템(MIRIX)을 제안.
2. **Active Retrieval 메커니즘**: 사용자가 명시적으로 메모리 검색을 요청하지 않아도 자동으로 관련 메모리를 검색하고 시스템 프롬프트에 주입.
3. **ScreenshotVQA 벤치마크**: 최대 20,000장의 고해상도 스크린샷을 처리하는 새로운 멀티모달 벤치마크를 제안 — 기존 메모리 시스템이 적용 불가능한 규모.
4. **LOCOMO SOTA**: LOCOMO 벤치마크에서 85.38%의 SOTA 성능 달성, 기존 최고 성능 대비 8.0%p 향상.
5. **크로스 플랫폼 애플리케이션**: React-Electron 기반 크로스 플랫폼 개인 비서 애플리케이션을 구현하여 실시간 스크린 모니터링 및 메모리 구축 기능 제공.

## Experiment & Results

**ScreenshotVQA 벤치마크** (3명 PhD 학생의 실제 스크린샷 데이터):
- MIRIX 전체 정확도 **59.50%** vs SigLIP@50 44.10% vs Gemini 11.66%.
- RAG 베이스라인(SigLIP) 대비 **35% 정확도 향상** + **99.9% 저장 공간 절감** (15.89MB vs 15.07GB).
- Long-context 베이스라인(Gemini) 대비 **410% 정확도 향상** + **93.3% 저장 공간 절감** (15.89MB vs 236.70MB).

**LOCOMO 벤치마크** (장문 다중 턴 대화, gpt-4.1-mini 백본):
- MIRIX 전체 정확도 **85.38%** — 전체 SOTA 달성.
- Single-Hop **85.11%**, Multi-Hop **83.70%**, Open-Domain **65.62%**, Temporal **88.39%**.
- 최강 오픈소스 경쟁자 LangMem(78.05%) 대비 **+7.33%p**, Zep(79.09%) 대비 **+6.29%p**.
- Multi-Hop에서 가장 큰 격차: MIRIX 83.70% vs Zep 69.16% (**+14.54%p**) — 통합된 에피소드 저장이 분산된 정보 연결 필요성을 제거.
- Full-Context 상한선(87.52%)과의 격차 단 **2.14%p** — 메모리 시스템만으로 거의 전체 맥락 수준에 도달.

## Limitation

- **Open-Domain 질문에서 상대적 약세**: MIRIX의 Open-Domain 정확도(65.62%)는 Full-Context(71.88%) 대비 6.26%p 낮으며, 이는 RAG 기반 검색이 글로벌 이해(global understanding)에서 본질적 한계를 가지기 때문이다.
- **Single-Hop 질문의 모호성 취약점**: 계획된 날짜 vs 실제 발생 날짜 등 모호한 질문에서 MIRIX가 통합 저장된 이벤트를 우선시하여 원래 의도와 불일치하는 경우가 발생한다.
- **평가 규모의 제한**: ScreenshotVQA 벤치마크는 3명의 사용자와 총 87개 질문으로 구성되어 통계적 일반화에 한계가 있다.
- **모델 의존성**: 멀티에이전트 구조상 각 에이전트가 다수의 함수 호출을 수행해야 하므로, 백본 모델의 function calling 능력에 크게 의존한다.
- **비공개 모델 전용**: 기존 latent-space 메모리 방식과 달리 모델 재훈련 없이 작동하나, 반대로 강력한 폐쇄형 모델(GPT-4, Gemini)에 의존하여 오픈소스 환경에서의 재현성이 제한적이다.
