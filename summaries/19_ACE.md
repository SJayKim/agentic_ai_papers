# ACE: Agentic Context Engineering — Evolving Contexts for Self-Improving Language Models

> **논문 정보**: Qizheng Zhang, Changran Hu, Shubhangi Upasani 외 (Stanford University, SambaNova Systems, UC Berkeley)
> **arXiv**: 2510.04618v2 (2026.01) | **학회**: ICLR 2026
> **코드**: https://github.com/ace-agent/ace

---

## Problem

LLM 기반 에이전트와 도메인 특화 추론 시스템은 모델 가중치 수정 대신 컨텍스트 적응(context adaptation)에 점점 더 의존하고 있다.

시스템 프롬프트, 메모리, 외부 증거 형태로 제공되는 컨텍스트는 모델의 실시간 행동과 도메인 적응을 좌우하는 핵심 자산이 되고 있다.

그러나 기존 컨텍스트 적응 방법들은 두 가지 구조적 한계에 부딪힌다.

첫째, **간결성 편향(brevity bias)** — GEPA, MIPROv2 같은 프롬프트 최적화기는 간결하고 범용적인 지시문을 선호하도록 설계되어, 도메인 특화 휴리스틱, 도구 호출 가이드라인, 반복적으로 관찰된 실패 모드 같은 실전에서 결정적인 세부 전략을 누락시킨다.

둘째, **컨텍스트 붕괴(context collapse)** — LLM이 축적된 컨텍스트를 단일 블록으로 엔드투엔드 재작성할 때, 점점 짧고 정보가 빈약한 요약으로 퇴화하여 성능이 급락한다.

저자들은 AppWorld 사례 연구에서 컨텍스트가 18,282 토큰(정확도 66.7%)이었던 상태에서 바로 다음 적응 단계에 122 토큰(정확도 57.1%)으로 붕괴하여 컨텍스트를 아예 사용하지 않는 기준선(63.7%)보다도 낮아지는 현상을 실증했다.

이 두 문제는 단일 모놀리식 프롬프트 업데이트에 의존하는 기존 패러다임이 도메인 지식을 누적·진화시키기에 근본적으로 부적합함을 드러낸다.

## Motivation

저자들은 컨텍스트가 간결한 일반 지침이 아니라 포괄적이고 진화하는 **플레이북(evolving playbook)** 으로 기능해야 한다고 주장한다.

인간과 달리 LLM은 길고 상세한 컨텍스트가 주어질 때 더 효과적으로 작동하며, 추론 시점에 자율적으로 관련성을 추출할 수 있는 능력을 갖추고 있다.

따라서 도메인 특화 전략·전술·실패 사례를 압축하여 지우기보다는, 가능한 한 보존하되 모델이 추론 시점에 필요한 항목만 선별하도록 하는 설계가 바람직하다.

가중치 대신 컨텍스트를 수정하는 방식은 해석 가능하고, 런타임에 새 지식을 빠르게 통합할 수 있으며, 컴포넌트 간 공유가 용이하다는 시스템 레벨 장점이 있다.

Long-context LLM의 급속한 발전과 KV 캐시 재사용·압축·오프로딩 같은 효율적 추론 기술의 등장으로 긴 컨텍스트 기반 접근이 실용화 단계에 도달했다.

Dynamic Cheatsheet가 제안한 테스트 시점 외부 메모리 아이디어는 옳은 방향이지만, 단일 프롬프트를 반복 재작성하는 방식은 앞서 지적한 두 한계를 그대로 겪는다.

따라서 저자들은 "생성-반성-큐레이션"을 명확히 분리한 에이전틱 아키텍처와 구조화된 항목(bullet) 기반 컨텍스트 표현을 결합하여, 정보 손실 없이 점진적으로 성장하는 컨텍스트를 만드는 방향을 제안한다.

## Method

ACE는 오프라인(시스템 프롬프트 최적화)과 온라인(테스트 시점 메모리 적응) 양쪽 모두에서 동일한 프레임워크로 컨텍스트를 진화시키는 모듈형 시스템이다.

1. **세 역할의 에이전틱 아키텍처** — Generator는 추론 궤적을 생성하고, Reflector는 성공/실패 궤적을 분석해 구체적 교훈을 추출하며, Curator는 교훈을 구조화된 델타 항목으로 합성하여 기존 컨텍스트에 병합한다. 세 역할은 모두 동일 LLM(DeepSeek-V3.1 non-thinking)을 공유하여 강한 Reflector/Curator가 약한 Generator로 지식을 유출하는 실험적 잡음을 차단한다.

2. **Reflector의 독립 분리** — Dynamic Cheatsheet가 평가와 큐레이션을 섞었던 구조를 해체해, 평가·교훈 추출을 Curator에서 떼어낸 전담 Reflector로 분리한다. Reflector는 여러 라운드에 걸쳐 자신의 교훈을 반복 정제할 수 있으며 최대 5 라운드까지 허용된다.

3. **구조화된 항목(bullet) 기반 컨텍스트 표현** — 컨텍스트를 단일 모놀리식 프롬프트가 아닌 개별 항목의 컬렉션으로 표현한다. 각 항목은 (a) 고유 식별자와 "도움됨/해로움" 카운터를 담는 메타데이터, (b) 재사용 가능한 전략·도메인 개념·실패 모드를 담는 콘텐츠로 구성된다.

4. **세 가지 운영 속성** — 이 표현은 (i) 항목 단위의 지역적 업데이트, (ii) 세밀한 검색으로 Generator가 가장 관련 있는 항목에만 집중, (iii) 증분적 적응을 가능하게 한다.

5. **증분적 델타 업데이트** — 전체 컨텍스트를 재생성하는 대신 Reflector가 추출한 소규모 후보 항목 집합을 Curator가 기존 컨텍스트에 증분 병합한다. 이는 모놀리식 재작성의 레이턴시·토큰 비용·정보 손실을 모두 회피한다.

6. **성장 및 정제 메커니즘(Grow-and-Refine)** — 신규 항목은 추가하고 기존 항목은 카운터만 갱신한다. 시맨틱 임베딩 비교로 유사 항목을 중복제거하며, 정제는 모든 업데이트마다 사전적으로 수행하거나 컨텍스트 윈도우 초과 시에만 지연 수행하도록 선택 가능하다.

7. **비-LLM 기반 결정론적 병합** — 델타 항목의 병합 자체는 LLM 호출 없이 경량 규칙 기반 로직으로 수행되어 비용이 무시할 수준이며, 여러 델타를 병렬 병합할 수 있다.

8. **배치 1 + 멀티 에폭 적응** — 배치 크기 1(샘플당 델타 구성)로 세밀한 업데이트를 보장하고, 동일 쿼리 집합을 여러 에폭(최대 5) 재방문하여 컨텍스트를 점진적으로 강화한다.

9. **오프라인/온라인 공용 파이프라인** — 동일한 Generator-Reflector-Curator 루프가 오프라인에서는 훈련 질의 위에서 시스템 프롬프트를 최적화하고, 온라인에서는 테스트 시점 메모리를 실시간 갱신한다.

10. **라벨 없는 적응(No-GT 모드)** — AppWorld의 코드 실행 성공/실패 같은 자연적 실행 피드백만으로도 Reflector가 유용/유해 신호를 뽑아낼 수 있어 지도 라벨 없이 동작한다.

11. **워밍업 결합** — 온라인 적응 전에 오프라인 워밍업으로 초기 플레이북을 구축하면 온라인 성능이 추가로 향상된다.

## Key Contribution

1. **두 핵심 실패 모드 진단** — 기존 컨텍스트 적응 방법의 간결성 편향과 컨텍스트 붕괴 현상을 정식화하고, 토큰 수/정확도 쌍으로 붕괴를 실증(18,282→122 토큰, 66.7%→57.1%).

2. **에이전틱 컨텍스트 엔지니어링 프레임워크** — Generator-Reflector-Curator 3역할 아키텍처 + 구조화된 항목 기반 컨텍스트 + 증분 델타 업데이트 + Grow-and-Refine을 통합한 ACE를 제안.

3. **Reflector의 평가-큐레이션 분리** — Dynamic Cheatsheet 대비 평가·교훈 추출 단계를 독립 컴포넌트로 떼어내 컨텍스트 품질과 학습 신호의 선명도를 높임.

4. **증분 델타 업데이트로 모놀리식 재작성 제거** — 로컬 편집만으로 과거 지식 보존과 신규 통찰 통합을 동시에 달성하고 컨텍스트 붕괴 방지.

5. **라벨 없는 자가 개선 입증** — 실행 피드백만으로도 AppWorld 정확도 +14.8% 향상, 외부 감독 없이 에이전트가 스스로 플레이북을 구축 가능.

6. **대폭적 비용·지연 절감** — 적응 레이턴시 평균 86.9% 감소, 롤아웃 수·토큰 비용 동시 감소로 확장 가능한 자기 개선 달성.

7. **소형 오픈소스 모델로 프로덕션급 성능** — DeepSeek-V3.1 + ACE가 AppWorld 리더보드 최상위 IBM CUGA(GPT-4.1 기반)와 평균 동등, test-challenge에서 상회.

## Experiment & Results

**평가 벤치마크** — AppWorld(자율 에이전트 API 이해·코드 생성·환경 상호작용), FiNER(XBRL 금융 엔티티 라벨링), Formula(XBRL 구조화 파일에서 값 추출·수치 추론). 평가 지표는 AppWorld의 경우 공식 프로토콜의 TGC(Task Goal Completion)·SGC(Scenario Goal Completion), FiNER/Formula는 정확도.

**베이스 LLM** — DeepSeek-V3.1 non-thinking 모드(Generator/Reflector/Curator 공용).

**AppWorld 오프라인** — ReAct + ACE 평균 **59.4%**, 기준선 Base LLM 42.4% 대비 **+17.0%p**, GEPA 46.4% 대비 **+13.0%p**. GT 라벨 없이도 **57.2%** (+14.8%p) 달성.

**AppWorld 온라인** — ACE **59.5%**, Dynamic Cheatsheet 51.9% 대비 **+7.6%p**.

**금융 분석(오프라인)** — FiNER **78.3%** (+7.6%p), Formula **85.5%**, 평균 **81.9%** (Base 69.1% 대비 +12.8%p), GEPA 72.5% 대비 +9.4%p, ICL·MIPROv2·GEPA 평균 대비 +10.9%p.

**AppWorld 리더보드(2025-09-20)** — 프로덕션급 IBM CUGA(GPT-4.1 기반)와 평균 동등, test-challenge split에서 TGC **+8.4%p**, SGC **+0.7%p** 상회.

**비용/지연(오프라인 AppWorld)** — GEPA 대비 적응 레이턴시 53,898s → 9,517s(**-82.3%**), 롤아웃 수 1,434 → 357(**-75.1%**).

**비용/지연(온라인 FiNER)** — Dynamic Cheatsheet 대비 적응 레이턴시 **-91.5%**, 토큰 비용 **-83.6%**.

**평균 레이턴시 절감** — 전체 실험 평균 **86.9%** 감소.

**Ablation(AppWorld)** — Reflector + 멀티에폭 제거 시 55.1%(−4.3%p), 멀티에폭만 제거 시 56.8%(−2.6%p), 오프라인 워밍업 추가 시 온라인 56.1% → 59.5%(+3.4%p).

**컨텍스트 붕괴 사례** — 재작성 1단계 전후로 토큰 수 18,282 → 122, 정확도 66.7% → 57.1%로 컨텍스트 미사용(63.7%)보다 낮아짐.

## Limitation

Reflector가 궤적에서 의미 있는 교훈을 추출하지 못하거나 잘못된 원인을 지목할 경우, 누적된 항목이 노이즈로 작용해 오히려 성능을 저해할 수 있다.

자연적 실행 피드백(단위 테스트 결과, 코드 실행 성공/실패, 수식 정답성 등) 같은 신뢰할 수 있는 신호가 없으면 ACE의 자가 개선 루프가 제대로 작동하지 않으며, 잘못된 신호가 주어지면 컨텍스트가 체계적으로 오염될 위험이 있다.

간결한 고수준 지시로 충분히 해결되는 태스크에서는 ACE가 생성하는 상세 플레이북이 불필요한 오버헤드이거나 중복적이어서 ROI가 낮다.

긴 컨텍스트의 서빙 비용은 KV 캐시 재사용·압축·오프로딩 같은 현대적 최적화가 없는 환경에서는 선형적으로 증가하여 실배포 부담이 될 수 있다.

실험은 에이전트(AppWorld)와 금융 분석(FiNER/Formula) 두 도메인에 집중되어 있어, 코드 수정·의료·법률·개방형 대화 등 다른 도메인으로의 일반화는 추가 검증이 필요하다.

베이스 LLM이 DeepSeek-V3.1로 고정되어 있어, 모델 규모나 패밀리(예: 소형 모델, 폐쇄형 프론티어 모델)에 따른 ACE의 상대적 이득 변화는 충분히 규명되지 않았다.

Reflector 정제 라운드·에폭 수·유사도 임계치 등 하이퍼파라미터 민감도 분석이 제한적이어서 새 도메인 적용 시 튜닝 비용이 발생할 수 있다.

최종적으로 에이전트가 자기 자신의 실패 사례를 정확히 분류해야 한다는 전제가 깔려 있어, 자기 평가 능력이 약한 모델에서는 프레임워크 전체의 유효성이 약화될 수 있다.
