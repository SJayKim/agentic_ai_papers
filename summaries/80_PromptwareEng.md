# Promptware Engineering: Software Engineering for Prompt-Enabled Systems

> **논문 정보**: Zhenpeng Chen (Tsinghua University), Chong Wang, Weisong Sun (Nanyang Technological University), Xuanzhe Liu (Peking University), Jie M. Zhang (King's College London), Yang Liu (Nanyang Technological University)
> **arXiv**: 2503.02400 (2025.03, v2 2026.01)
> **출간 예정**: ACM Trans. Softw. Eng. Methodol. (2026)
> **코드**: N/A (비전 페이퍼)

---

## Problem
- LLM이 GPT, LLaMA, DeepSeek 등을 중심으로 Microsoft, Google, Amazon, Apple 같은 주요 기업의 제품에 광범위하게 통합되면서, 프롬프트가 시스템 동작을 제어하는 핵심 프로그래밍 인터페이스로 부상하였다.
- 이러한 프롬프트 기반 시스템(prompt-enabled systems)에서 자연어 프롬프트를 최상위 소프트웨어 아티팩트로 다루는 "promptware" 패러다임이 등장하였다.
- 그러나 현재 프롬프트 개발은 체계적 방법론 없이 ad hoc 방식과 시행착오(trial-and-error)에 크게 의존하고 있으며, 저자들은 이를 "promptware crisis"로 명명한다.
- 기존 프롬프트 엔지니어링 가이드라인(OpenAI, Meta 등)은 출력 중심(output-centric)이며, 개발 프로세스 전반을 아우르는 체계적 방법론이 부재하다.
- 경험 많은 엔지니어조차 프롬프트 개발에 어려움을 겪으며, Tafreshipour et al.의 실무 연구에 따르면 실제 프로젝트에서 프롬프트 변경의 21.9%만이 문서화되고 있다.
- 프롬프트는 비정형 자연어로 작성되어 모호하고 맥락 의존적이며, LLM 런타임은 확률적·비결정적이어서 기존 소프트웨어와 본질적으로 다른 고유한 도전 과제를 야기한다.
- LangChain, Liquid 같은 템플릿 기반 접근법이 등장했으나, 이들은 프롬프트 오케스트레이션과 텍스트 템플릿 수준에 머물러 언어 수준의 추상화를 제공하지 못한다.
- Mailach et al., Parnin et al., Dolata et al. 등 다수의 실증 연구는 프롬프트 엔지니어링이 시간 소모적이고 자원 제약적이며 예측 불가능하다는 공통된 결론에 도달한다.

## Motivation
- 수십 년간 축적된 소프트웨어 공학(SE)의 체계적 방법론이 복잡성 관리, 품질 보증, 반복적 개선에 성공적으로 활용되어 왔다.
- 프롬프트가 소프트웨어의 핵심 구성요소로 자리잡으면서, 프롬프트 개발을 프로그래밍의 새로운 형태로 인식해야 할 필요성이 대두된다.
- 기존 연구들은 SE를 위한 프롬프트 엔지니어링(prompt engineering for SE)에 집중했지, 프롬프트 개발을 위한 SE 관점(SE for prompt engineering)은 거의 탐구되지 않았다.
- 자연어의 비정형성, 모호성, 비결정적 실행, 블랙박스 특성, 에러 핸들링 부재 등 10가지 고유 특성은 기존 SE 기법의 직접 적용을 어렵게 만든다.
- 이러한 gap을 해소하기 위해, SE 원칙을 프롬프트 개발 전 라이프사이클에 체계적으로 적용하는 "promptware engineering" 프레임워크를 제안한다.
- 컴퓨터 과학, 언어학, 심리학 등 학제간 협력을 통해 프롬프트 개발을 실험적 관행에서 구조적이고 규율 있는 프로세스로 전환하고자 한다.
- Hassan et al.의 선행 비전(프롬프트 인식 IDE, 프롬프트 트랜스파일러)이 개별 도구 수준에 머물렀던 반면, 본 논문은 전체 라이프사이클 관점의 포괄적 프레임워크를 제시한다.
- 프롬프트웨어 엔지니어링은 전통 SE를 대체하는 것이 아니라, LLM 시대의 자연스러운 진화로서 요구공학·설계·테스팅·배포 등 기존 활동을 재해석하는 접근이다.

## Method
1. **Promptware 고유 특성 분석 (C1-C10)**
   - 언어 차원: 구조/형식성(C1), 명시성/결정성(C2), 정확성/품질(C3)의 3가지 특성을 전통 프로그래밍 언어와 비교 분석
   - 런타임 차원: 결정성(C4), 인간유사 특성(C5), 능력 경계(C6), 에러 핸들링(C7), 실행 제어(C8), 접근 제어(C9), 메모리 관리(C10)의 7가지 특성을 전통 런타임과 비교 분석
   - 각 특성이 프롬프트 개발에 미치는 구체적 도전 과제를 체계적으로 도출하며, Table 1(언어)과 Table 2(런타임)로 정리
   - 예: C4는 동일 프롬프트가 서로 다른 출력을 생성하는 재현성 문제, C9는 프롬프트 인젝션·민감 정보 유출 같은 보안 취약점

2. **8대 SE 활동에 대응하는 프롬프트웨어 엔지니어링 로드맵 (27개 연구 기회)**
   - **Prompt Requirements Engineering (O1-O4)**: LLM 능력 기반 요구사항 정의, 기능/비기능 요구사항 구분, 다목적 트레이드오프 분석, 모호성 내성 명세(semi-formal specification language) 개발
   - **Prompt Design (O5-O8)**: 형식화된 프롬프트 디자인 패턴 정립(zero-shot/few-shot/CoT/ToT/self-consistency/RAG를 체계적으로 분류), UML 유사 설계 도구 개발, 응집도/결합도/복잡도 기반 설계 메트릭 제안, 패턴 리포지토리 구축
   - **Prompt Implementation (O9-O15)**: 프롬프트 중심 프로그래밍 언어 설계(CNL-P 등 선언적 언어 확장), 프롬프트 컴파일(어휘·구문분석→중간표현→의미최적화→토큰압축→보안강화), 프롬프트 전용 IDE, 검색 기반 최적화(유전 알고리즘), 프롬프트 라이브러리/API 표준화, role-playing 전략, 온라인 프롬프트 구현
   - **Prompt Testing & Debugging (O16-O22)**: flaky test 해결(80% 성공 임계치 등), 테스트 입력 생성, 메타모픽 테스팅 기반 테스트 오라클, 테스트 적절성 메트릭, 단위·통합 테스팅, 비기능 테스팅(공정성·보안·프라이버시), 세션 기반 디버깅 프레임워크
   - **Prompt Evolution (O23-O24)**: 코드/LLM/사용자 피드백 기반 진화 관리, 의미적 버전 관리(semantic versioning) 및 추적성 확보
   - **Prompt Deployment & Monitoring (O25-O27)**: CI/CD 유사 배포 파이프라인, 카나리 배포/단계적 릴리스, 런타임 모니터링/관측성, 피드백 루프 기반 적응적 모니터링

3. **Illustrative Vignette**: 고객 지원 시스템 사례를 통해 요구사항→설계→구현→테스팅→배포→모니터링→진화의 전체 라이프사이클을 예시적으로 시연
4. **C-O 매핑 매트릭스**: 각 연구 기회(O1-O27)에 관련된 promptware 특성(C1-C10)을 명시적으로 연결하여, 어떤 고유 도전을 해결하는지 투명하게 제시

## Key Contribution
1. "Promptware"의 10가지 고유 특성(C1-C10)을 체계적으로 식별하고, 전통 소프트웨어 패러다임과 언어·런타임 두 축에서 비교 분석하였다.
2. 프롬프트 개발의 ad hoc 관행을 "promptware crisis"로 정의하고, 이를 해결하기 위한 체계적 프레임워크의 필요성을 학술적으로 공식화하였다.
3. SE 라이프사이클 8개 활동에 걸쳐 27개 구체적 연구 기회(O1-O27)를 제시하는 포괄적 로드맵을 수립하였다.
4. 프롬프트 컴파일, 프롬프트 중심 프로그래밍 언어, 프롬프트 디자인 패턴 등 기존 SE 개념의 프롬프트 도메인 적응 방안을 구체적으로 제안하였다.
5. SE와 AI 커뮤니티 간 학제적 협력의 필요성을 강조하며, 프롬프트웨어 엔지니어링을 독립적 연구 분야로 정립할 비전을 제시하였다.
6. 기존의 개별 도구 중심 접근(Hassan et al.의 prompt-aware IDE 등)을 넘어, 프롬프트를 완전한 소프트웨어 아티팩트로 다루는 전체 라이프사이클 관점을 정립하였다.

## Experiment & Results
- 본 논문은 비전 페이퍼(vision paper)로서, 실험적 검증이나 벤치마크 평가를 포함하지 않는다.
- 대신, 관련 실증 연구의 정량적 발견을 근거로 프레임워크의 필요성을 뒷받침한다.
- Tafreshipour et al.의 연구에 따르면, 실제 프로젝트에서 프롬프트 변경의 **21.9%**만이 문서화되어 관리되고 있다.
- 기존 연구에서 LLM 온도(temperature)를 0으로 설정해도 여전히 일정 수준의 출력 변동성이 존재함이 확인되었다.
- flaky test 대응으로 **80% 이상** 성공률 임계치 기반 합격 판정 방식이 대안으로 제안되었다.
- 로드맵에서 제시된 **27개 연구 기회**는 **10개 promptware 특성(C1-C10)**과 체계적으로 매핑되어 있다.
- 프롬프트 컴파일은 **5단계**(어휘/구문 분석, 중간 표현 생성, 의미 최적화, 토큰 압축, 보안 강화)로 구성된다.
- **8개 핵심 SE 활동** 영역(요구공학, 설계, 구현, 테스팅·디버깅, 진화, 배포·모니터링)에 걸쳐 프롬프트웨어 엔지니어링 프레임워크가 구조화되었다.
- Microsoft 제품팀 인터뷰(Nahar et al.)를 포함한 **6건 이상의 실증 연구**(Mailach, Parnin, Dolata, Liang, Tafreshipour, Nahar 등)가 프롬프트 개발 도전 과제를 뒷받침하는 근거로 인용되었다.
- 프롬프트 설계 품질 평가를 위한 **4가지 메트릭 계열**(cohesion, coupling, complexity, memory/probabilistic determinism/error-handling)이 제안되었다.
- 프롬프트 테스팅에서 LLM-as-a-judge, 메타모픽 테스팅, 다중 검토자 합의(multi-reviewer consensus) 등 **3가지 이상 테스트 오라클** 대안이 제시되었다.

## Limitation
- 제안된 프레임워크는 이론적·개념적 수준에 머물러 있으며, 실제 프롬프트 기반 시스템에서의 구현이나 경험적 검증이 수행되지 않았다.
- 프레임워크의 실용적 효과성, 확장성, 사용성은 아직 실증되지 않았으며, 체계적 사례 연구나 배포 시험은 향후 과제로 남아 있다.
- 27개 연구 기회 각각에 대한 구체적 구현 방안이나 도구 설계는 제시되지 않아, 실행 가능한 기술적 세부사항이 부족하다.
- LLM의 급속한 발전 속도를 고려할 때, 제안된 프레임워크가 지속적으로 유효할 수 있는지에 대한 적응성 논의가 제한적이다.
- 프롬프트 디자인 패턴, 프롬프트 컴파일 등 핵심 개념들이 아직 프로토타입 수준의 검증도 이루어지지 않아, 개념과 실제 간의 gap이 존재한다.
- 소규모 팀이나 리소스 제한 조직에서는 SE와 프롬프트 엔지니어링 양쪽의 전문성이 필요하여 채택이 어려울 수 있다.
- 본 논문은 SE 커뮤니티의 관점에 크게 의존하며, AI/ML 커뮤니티의 기존 프롬프트 연구 방법론과의 통합 방안이 충분히 논의되지 않았다.
- 멀티모달 프롬프트(이미지, 음성, 코드 혼합)에 대한 고려가 부족하며, 텍스트 프롬프트에 초점이 맞춰져 있다.
- 프롬프트 중심 프로그래밍 언어의 구체적 문법이나 컴파일러 구현체는 제시되지 않았으며, 언어 설계 원칙만 서술되었다.
- 프롬프트웨어 특유의 라이선스·저작권 문제(O15에서 언급)에 대한 해결 방안이 구체화되지 않아, 생태계 확장 시 법적·윤리적 리스크가 남아 있다.
- 27개 연구 기회 간의 의존성·우선순위·실행 순서에 대한 분석이 부재하여, 실제 연구자들이 어디서부터 시작해야 할지 실천적 가이드가 제한적이다.
