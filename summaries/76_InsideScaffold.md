# Inside the Scaffold: A Source-Code Taxonomy of Coding Agent Architectures

> **논문 정보**: Benjamin Rombaut (Centre for Software Excellence, Huawei Canada)
> **arXiv**: 2604.03515 (2026.04)
> **코드**: N/A

---

## Problem
- LLM 기반 코딩 에이전트들이 버그 수정, 패치 생성, 테스트 실행 등을 자동화하고 있으나, LLM을 둘러싼 스캐폴딩 코드(제어 루프, 도구 정의, 상태 관리, 컨텍스트 전략)에 대한 체계적 이해가 부족하다.
- 기존 서베이는 에이전트를 추상적 능력(도구 사용, 계획, 반성) 기준으로 분류하여, 아키텍처적으로 근본적으로 다른 시스템을 구별하지 못한다.
- 예를 들어, MCTS로 패치를 탐색하는 에이전트와 단순 while 루프로 테스트 기반 재시도를 하는 에이전트가 모두 "도구 사용, 계획, 반성" 에이전트로 동일하게 분류된다.
- 트라젝토리 연구들은 에이전트가 "무엇을" 하는지는 관찰하지만, 스캐폴드 코드가 "왜" 그렇게 행동하는지는 설명하지 못하는 블랙박스 접근을 취한다.
- 선행 연구들이 서로 다른 LLM을 사용한 에이전트를 비교하여, 행동 차이가 스캐폴드에서 비롯된 것인지 모델에서 비롯된 것인지 분리가 불가능하다.
- 실무자들은 새로운 코딩 에이전트를 설계할 때 참고할 체계적 설계 공간 맵이 없어, 개별 코드베이스나 블로그를 읽어야 하는 상황이다.
- 13개 에이전트가 2023년 6월~2025년 3월에 걸쳐 출시되었으나, 이들의 아키텍처적 관계를 매핑한 연구는 존재하지 않았다.

---

## Motivation
- 기존 능력 기반 분류 체계(tool-using, memory-augmented, planning 등)로는 프로덕션 코딩 에이전트 간의 실질적 차이를 표현할 수 없다.
- 스캐폴드 설계와 모델 능력 사이의 교란 변수(confound)가 인식조차 되지 않고 있어, 연구자들이 관찰된 차이를 특정 아키텍처 선택에 귀인할 수 없다.
- 소스 코드 수준에서 에이전트 스캐폴드를 비교 분석하면, 행동 차이의 설명적 계층(explanatory layer)을 제공할 수 있다.
- 아키텍처 인식 평가 메트릭(architecture-aware evaluation metrics) 구축을 위한 어휘와 변수 정의가 필요하다.
- 저자는 구현 수준에서 관찰된 패턴으로부터 분류 체계를 도출하는 질적 사례 연구(qualitative case study) 방법론을 채택했다.
- 오픈 소스 에이전트 13개를 고정된 커밋 해시에서 분석하여, 재현 가능하고 검증 가능한 증거 기반을 구축하고자 했다.
- 이를 통해 연구자에게는 행동 연구의 설명 도구를, 실무자에게는 새 스캐폴드 설계 시 비교 참조를 제공하는 것이 목표이다.

## Method
**분석 대상 선정**: SWE-bench 리더보드, GitHub 스타 수, 최근 서베이 인용 3개 채널에서 후보 22개를 식별하고, 코딩 특화·오픈소스·아키텍처 고유성 3개 기준으로 13개를 선정했다.
- CLI 에이전트 7개: OpenCode(135k stars), Gemini CLI(100k), Codex CLI(72k), Cline(60k), Aider(43k), SWE-agent(19k), mini-swe-agent(4k)
- SWE-bench 에이전트 6개: OpenHands(70k), AutoCodeRover(3k), Agentless(2k), Prometheus(1k), Moatless Tools(600), DARS-Agent(70)

**분석 프레임워크 (3계층 12차원)**:
- **Layer 1 - 제어 아키텍처**: (1) 제어 루프 전략, (2) 루프 드라이버, (3) 제어 흐름 구현
- **Layer 2 - 도구 및 환경 인터페이스**: (4) 도구 세트 설계, (5) 편집/패치 포맷, (6) 도구 발견 전략, (7) 컨텍스트 검색 패러다임, (8) 실행 격리
- **Layer 3 - 리소스 관리**: (9) 상태 관리, (10) 컨텍스트 압축, (11) 다중 모델 라우팅, (12) 영속 메모리

**5가지 루프 프리미티브**: ReAct, generate-test-repair, plan-execute, multi-attempt retry, tree search — 이들은 상호 배타적 유형이 아닌 조합 가능한 빌딩 블록으로 기능한다.

**제어 루프 스펙트럼**: 고정 파이프라인(Agentless) → 사용자 구동 루프(Aider) → 순차 ReAct 루프(7개 에이전트) → 단계별 루프(AutoCodeRover, Prometheus) → 깊이 우선 트리 탐색(DARS-Agent) → 전체 MCTS(Moatless Tools)

**방법론**: Aider(단순)와 OpenHands(복잡)를 파일럿 분석하여 초기 6개 차원을 12개로 확장. 각 에이전트에 대해 관찰(코드 동작)→분류(차원 매핑)→증거(파일 경로·라인 번호) 3단계 구조화 템플릿 적용. 오픈 코딩 방식으로 47개 교차 패턴 발견.

**도구 카운트 이중 방법론**: 등록 카운트(LLM이 보는 개별 호출 단위)와 능력 범주 카운트(read, search, edit, execute, validate)를 병행 보고.

## Key Contribution
1. 13개 오픈소스 코딩 에이전트 스캐폴드의 소스 코드 분석에서 도출된, 3계층 12차원의 아키텍처 분류 체계를 최초로 제시했다.
2. 스캐폴드 아키텍처가 이산적 유형이 아닌, 루프 프리미티브의 조합으로 이루어진 연속 스펙트럼으로 더 잘 특성화된다는 실증적 발견을 보고했다.
3. 5가지 루프 프리미티브(ReAct, generate-test-repair, plan-execute, multi-attempt retry, tree search)를 조합 가능한 빌딩 블록으로 식별하고, 13개 중 11개(85%)가 복수 프리미티브를 조합함을 밝혔다.
4. 수렴 차원과 발산 차원을 구분: 도구 능력 범주, 편집 포맷, 실행 격리는 수렴하고, 컨텍스트 압축(7가지 전략), 상태 관리, 다중 모델 라우팅은 발산한다.
5. 특정 커밋에 고정된 파일 경로와 라인 번호로 뒷받침되는 상세 증거 기반을 제공하여, 독립적 검증과 재사용이 가능한 참조 자료를 구축했다.
6. 향후 제어된 실험 설계를 위한 변수(동일 도구 세트 + 다른 루프 전략 등)를 명시적으로 식별했다.

## Experiment & Results
본 연구는 벤치마크 성능 평가가 아닌 순수 분류학적(taxonomic) 분석이며, 소스 코드 관찰에서 다음의 정량적·정성적 결과를 도출했다.
- **제어 루프**: 13개 중 7개(54%)가 순차 ReAct 루프를 주 제어 구조로 사용하며, 11개(85%)가 복수 루프 프리미티브를 조합한다.
- **도구 세트**: 도구 수는 0개(Aider)에서 37개 액션 클래스(Moatless Tools)까지 분포하나, 4가지 핵심 능력 범주(read, search, edit, execute)가 모든 LLM 구동 에이전트에서 수렴한다.
- **편집 포맷**: str_replace_editor 방식이 독립적으로 개발된 5개 에이전트에서 공통 발견. Aider는 13가지 모델별 편집 포맷을 지원하며, SWE-agent는 10가지 출력 파서를 제공한다.
- **컨텍스트 압축**: 7가지 뚜렷한 전략이 관찰됨 — 무관리, 규칙 기반 절삭, 구조적 격리, 토큰 기반 선택적 포함, LLM 요약(5개 에이전트), LLM 요약+검증(Gemini CLI), LLM 주도 압축(Cline).
- **다중 모델 라우팅**: Codex CLI가 4개 모델로 최다. Gemini CLI는 7계층 우선순위 라우팅 체인 보유.
- **영속 메모리**: 벤치마크 에이전트 5개는 세션 간 메모리 없음. CLI 에이전트 5개가 프로젝트 규약 기억을 구현. Prometheus는 Neo4j + PostgreSQL + Athena 3계층 영속성.
- **검증**: 296개 추출 주장 중 267개 확인, 19개 수정, 10개 경미한 단순화로 수용.
- **생태계 성숙도**: MCP를 5개 에이전트(OpenHands, Codex CLI, Gemini CLI, Cline, OpenCode)가 지원.

## Limitation
- **단일 저자 편향**: 모든 분석이 한 명의 연구자에 의해 수행되어, 차원 분류 및 증거 선택에 주관성이 존재한다.
- **오픈소스 한정**: Claude Code, GitHub Copilot Workspace, Cursor, Windsurf 등 독점 에이전트가 제외되어, 전체 설계 공간을 대표하지 못한다.
- **정적 분석의 한계**: 소스 코드 정적 분석만 수행하여, MCP 도구 발견이나 런타임 행동은 포착하지 못한다.
- **스냅샷 특성**: 고정 커밋 해시 기반이므로 활발히 개발 중인 에이전트의 현재 아키텍처를 반영하지 않을 수 있다.
- **프롬프트 엔지니어링 미포함**: 시스템 프롬프트 구성, 길이, few-shot 예시 등 프롬프트 전략 차원이 범위에서 의도적으로 제외되었다.
- **Python 편향**: 13개 중 10개가 Python 구현이고, SWE-bench가 Python 프로젝트만 사용하여 다중 언어 생태계의 아키텍처 변이를 충분히 반영하지 못한다.
