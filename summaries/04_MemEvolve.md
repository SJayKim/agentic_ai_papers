# MemEvolve: Meta-Evolution of Agent Memory Systems

> **논문 정보**: Guibin Zhang, Haotian Ren, Chong Zhan, Zhenhong Zhou, Junhao Wang, He Zhu, Wangchunshu Zhou, Shuicheng Yan (OPPO AI Agent Team, LV-NUS lab)
> **arXiv**: 2512.18746v1 (2025.12.21) | **학회**: -
> **코드**: https://github.com/bingreeky/MemEvolve

---

## Problem

기존 LLM 에이전트의 자기 개선(self-improving) 메모리 시스템은 **수작업으로 설계된 고정 메모리 아키텍처** 위에서 경험만을 축적하는 구조로, 메모리 인터페이스 Ω 자체는 한 번 설계되면 변하지 않는다.

이 정적 특성은 다양한 태스크 도메인에 대한 적응을 원천적으로 차단하여, 특정 태스크에서 효과적인 설계가 다른 태스크에서는 오히려 성능을 저하시키는 근본적 불일치를 초래한다.

예를 들어 궤적에서 재사용 가능한 API를 추출하는 메모리는 웹 브라우징에서는 유효하지만 수학·과학 추론에는 거의 무용하고, 자기 비판(self-critique) 기반 메모리는 추론 집약적 태스크에는 강하지만 코딩·도구 사용 시나리오에서 효과가 뚜렷이 떨어진다.

ExpeL처럼 ALFWorld나 HotpotQA 등 단순 embodied/QA 환경용 프롬프트로 설계된 시스템은 장기 심층 연구 벤치마크에서 실제로 No-Memory 기준선보다도 낮은 점수를 기록한다.

DILU 같은 시스템도 xBench·WebWalkerQA에서는 향상되지만 GAIA에서는 −2.42%로 저하되는 등, 단일 메모리 설계가 여러 벤치마크에서 일관되게 성능을 올리지 못한다.

연구자들은 일반적으로 메모리 주입/추상화/검색 파이프라인을 고정해 두고 새로운 경험에만 노출시키면 장기 진화가 일어날 것이라 가정하지만, 태스크마다 요구하는 메모리 affordance가 다르다는 현실을 간과한다.

즉 **메모리 아키텍처의 정적 특성 자체가 범용 에이전트 자기 진화의 근본 병목**이며, 이는 경험 축적만으로는 극복되지 않는 설계 수준의 한계이다.

이러한 한계는 "어떤 메모리 아키텍처가 에이전트 자기 개선에 가장 효과적인가"라는 질문에 보편적 정답이 없음을 의미하므로, 태스크 환경에 맞춰 메모리 아키텍처 자체가 재조정되어야 한다.

## Motivation

인간 학습자의 비유가 핵심 직관을 제공한다 — 단순히 경험을 축적하는 "숙련 학습자(skillful learner)"와, 학습 전략 자체를 과목·상황에 따라 동적으로 조정하는 "적응 학습자(adaptive learner)"는 근본적으로 다르다.

문학 분석에서는 암기를, 수학에서는 풀이 템플릿 추상화를 우선하는 식으로 학습 전략(메타 전략)을 바꾸는 것이 진정한 적응이며, 우수한 학습자일수록 적응적 메타 학습 능력을 갖추고 있다.

에이전트 메모리 시스템도 마찬가지로 경험 축적(1차 진화)뿐 아니라 **메모리 아키텍처 자체의 메타 진화(2차 진화)**를 동시에 수행해야 한다는 관찰이 이 연구의 출발점이다.

이를 이중 수준 최적화(bilevel optimization)로 정형화하면, 내부 루프에서는 고정된 메모리 시스템 아래 경험을 쌓고, 외부 루프에서는 그 성능 피드백을 바탕으로 메모리 아키텍처를 진화시키는 선순환(virtuous cycle)을 구축할 수 있다.

이 선순환에서는 외부 루프가 개선한 아키텍처가 내부 루프의 학습 효율을 높이고, 더 유능해진 에이전트가 더 높은 품질의 궤적을 생성하여 외부 루프에 더 정확한 적합도 신호를 제공한다.

그러나 지식 그래프, 스킬 라이브러리, 벡터 DB 등 메모리 설계 공간은 매우 이질적이고 방대하므로 통제 가능한 최적화를 위해서는 **모듈형 분해**가 필수적이라는 설계 제약이 따른다.

따라서 저자들은 모든 메모리 시스템을 Encode/Store/Retrieve/Manage 네 개의 기능 모듈로 분해하여 표준 인터페이스를 부여하고, 이 네 모듈의 구현만 진화시키는 방식으로 탐색 공간을 다루기 쉽게 만든다.

## Method

1. **모듈형 메모리 설계 공간 정의**: 임의의 메모리 시스템 Ω를 네 가지 기능 모듈 Ω=(E, U, R, G)로 분해한다 — **Encode**(경험을 구조화 표현 e_t=E(ε_t)로 변환), **Store**(영속 메모리 M_{t+1}=U(M_t, e_t)에 통합), **Retrieve**(c_t=R(M_t, s_t, Q)로 관련 메모리 검색), **Manage**(통합·망각·정리).

2. **EvolveLab 통합 코드베이스 구축**: Voyager, ExpeL, Generative Agents, DILU, AWM, Mobile-E, Dynamic Cheatsheet, SkillWeaver, G-Memory, Agent-KB, Mem^p, EvolveR 등 12개 대표 메모리 시스템을 단일 추상 클래스 `BaseMemoryProvider` 아래 네 모듈 인터페이스로 재구현하여 표준화된 비교 플랫폼을 제공한다.

3. **이중 진화 프로세스 - 내부 루프(Experience Evolution)**: 각 후보 메모리 시스템 Ω_j^(k) 아래 에이전트가 태스크 배치 T_j^(k)를 실행하며 빈 메모리에서 출발해 경험을 축적하고, 각 궤적 τ마다 (task success, token consumption, latency) 3차원 피드백 벡터 f_j(τ)∈R^3을 수집한다.

4. **이중 진화 프로세스 - 집계**: 집계 연산자 S가 후보별 피드백을 요약하여 F_j^(k) = (Perf_j^(k), −Cost_j^(k), −Delay_j^(k)) 형태의 Pareto 벡터로 변환한다.

5. **이중 진화 프로세스 - 외부 루프(Architectural Evolution)**: 메타 진화 연산자 F가 후보 집합의 요약을 기반으로 상위 K개 아키텍처를 선택하고, 각 부모에서 S개의 변형 후손을 생성하여 다음 세대 후보 집합 {Ω_{j'}^(k+1)}을 구성한다.

6. **Pareto 기반 아키텍처 선택**: 성능·비용·지연의 3차원에 대해 비지배 정렬(non-dominated sorting)을 수행하여 Pareto 순위 ρ_j^(k)를 매기고, 동일 순위 내에서는 주요 성능 지표 Perf_j^(k)로 추가 정렬하여 상위 K(=1)개를 부모로 선택한다.

7. **Diagnose 단계**: 각 부모 아키텍처의 실행 궤적을 리플레이 인터페이스로 재검토하여 검색 실패, 비효율적 추상화, 저장 비효율 등 네 모듈별 병목을 드러내는 구조화된 결함 프로파일 D(Ω_p^(k))를 생성한다.

8. **Design 단계**: 결함 프로파일을 조건으로 삼아 모듈 인터페이스 내 허용된 구현 지점만을 수정하여 S(=3)개의 유효한 후손 변형 Ω_{p,s}^(k+1)을 생성하며, 각 변형은 인코딩 전략·저장 규칙·검색 제약·관리 정책이 서로 다르지만 통합 인터페이스를 준수한다.

9. **변형 예시**: 예컨대 부모 `AKB Mem`에서 Encode를 "9개 스킬 granularity 분해"로 바꾼 공격적 변형(Adaptive Decision)과 "4레벨 궤적 저장 + LLM 메타 가드레일"을 도입한 보수적 변형(Meta Mem)이 병렬로 생성된다.

10. **반복 진화**: 기본 설정에서 K_max=3 반복을 수행하며, 매 반복마다 각 후보에 대해 40개 새 태스크 + 이전 반복 20개 재사용 = 총 60개 태스크 궤적으로 평가하여 inter-iteration 비교를 안정화한다.

11. **최종 시스템 고정 및 전이**: 진화가 끝난 최적 메모리 아키텍처(예: Riva, Cerebra)를 고정한 뒤, 메타 진화에 쓰이지 않은 새 벤치마크와 새 LLM 백본으로 무수정 전이하여 일반화를 검증한다.

12. **에이전트 프레임워크 통합**: MemEvolve는 SmolAgent(경량 2-에이전트), Flash-Searcher(단일 에이전트), CK-Pro(3-에이전트), OWL(계층적 멀티 에이전트) 등 다양한 스캐폴딩에 플러그 앤 플레이로 통합되어 프레임워크 독립성을 확보한다.

## Key Contribution

1. **EvolveLab 통합 코드베이스**: 12개 대표 자기 개선 메모리 시스템을 Encode/Store/Retrieve/Manage의 4모듈 인터페이스로 재구현한 최초의 표준화된 실험 플랫폼으로, 기존에 산재했던 메모리 시스템들의 공정한 체계적 비교를 가능하게 하여 연구 커뮤니티의 재현성을 끌어올린다.

2. **메모리 아키텍처의 메타 진화 프레임워크**: 경험 축적(1차 진화)과 메모리 아키텍처 자체의 진화(2차 진화)를 이중 루프로 결합한 최초의 프레임워크로, 고정된 메모리 아키텍처가 다양한 태스크에서 일관된 향상을 주지 못한다는 구조적 한계를 메타 학습 방식으로 해소한다.

3. **Diagnose-and-Design 진화 연산자**: Pareto 기반 다목적 선택과 궤적 리플레이 진단을 결합하여, 성능·비용·지연의 균형을 유지하면서 모듈 단위로 제어 가능한 아키텍처 진화를 실현하고 무분별한 변형 폭발을 막는다.

4. **교차 태스크/모델/프레임워크 일반화 검증**: TaskCraft에서만 진화시킨 메모리를 WebWalkerQA, xBench, GAIA 및 이종 프레임워크(CK-Pro, OWL)에 무수정 전이하여 최대 17.06%의 성능 향상을 실증하며, 메모리 설계가 태스크 비의존적 원리를 포착함을 보였다.

5. **비용 중립적 성능 향상**: API 비용이 No-Memory 기준선과 사실상 동일한 상태에서 3.54~5.0%의 안정적 향상을 달성하여, "좋은 메모리는 반드시 비싸다"는 통념을 깨고 실용적 배포 가능성을 입증했다.

6. **진화된 아키텍처의 설계 원리 추출**: 자동 진화 결과를 분석하여 agentic한 의사결정, 계층적 조직화, 다단계 추상화라는 세 가지 공통 설계 원리를 추출하여 후속 메모리 시스템 수작업 설계에도 지침을 제공한다.

## Experiment

- **벤치마크**: GAIA, WebWalkerQA, xBench-DeepSearch, TaskCraft 4종을 사용하며, 메타 진화는 TaskCraft에서만 수행하고 나머지 3개는 전이 평가에 활용한다.

- **에이전트 프레임워크**: SmolAgent(경량 2-에이전트), Flash-Searcher(단일 에이전트), CK-Pro(3-에이전트), OWL(계층적 멀티 에이전트) 네 가지를 사용하여 아키텍처 다양성 아래 일반화를 검증한다.

- **LLM 백본**: GPT-5-Mini를 메타 진화·에이전트 구동에 사용하고, DeepSeek V3.2와 Kimi K2를 cross-LLM 일반화 검증용으로 추가한다.

- **Flash-Searcher + MemEvolve pass@1**: GAIA **73.33%** (No-Memory 69.09%, +4.24%p), xBench **74.0%** (69.0%, +5.0%p), WebWalkerQA **74.71%** (71.18%, +3.53%p)로 전 벤치마크 일관 향상.

- **Flash-Searcher + MemEvolve pass@3**: GAIA **80.61%**, xBench **78.0%**, WebWalkerQA **81.18%**로 OWL-Workforce·CK-Pro 등 강력한 멀티 에이전트 시스템을 상회.

- **SmolAgent + MemEvolve**: xBench pass@1 51%→57% (+6%), pass@3 68%로 경량 프레임워크에서도 뚜렷한 향상 확인.

- **교차 LLM 일반화**: GPT-5-Mini로 진화한 메모리를 무수정 전이했을 때 Kimi K2가 WebWalkerQA에서 **+17.06%**, TaskCraft에서 **+10.0%**의 큰 폭 향상 달성.

- **비용 효율**: API 비용이 No-Memory 기준선과 거의 동일 — GAIA $0.085 vs $0.086, xBench $0.136 vs $0.141, WebWalkerQA $0.040 vs $0.048.

- **실행 지연**: GAIA 693.33초로 AWM(584.88초)·Cheatsheet(559.81초)와 비슷한 수준을 유지하여 과도한 오버헤드 없음.

- **기존 메모리 시스템 비교**: Voyager, ExpeL, Generative, DILU, AWM, Mobile-E, Cheatsheet 등 7개 중 어느 것도 3개 벤치마크에서 일관된 향상을 달성하지 못함. ExpeL은 전 벤치마크에서 기준선 이하, DILU는 GAIA에서 −2.42%.

- **Ablation / 진화 궤적**: 3라운드에 걸쳐 공격적 변형(9-스킬 분해)과 보수적 변형(4-레벨 + LLM 가드레일)이 경쟁하다가, 텍스트 인사이트 + 재사용 도구 + 주기적 DB 유지보수를 갖춘 Cerebra로 수렴.

- **누적 정확도 분석**: 초기에는 샘플 수 부족으로 변동성이 크지만 태스크 수가 늘수록 MemEvolve가 다른 시스템보다 뚜렷이 높은 수준에서 안정화됨을 확인.

## Limitation

- **진화 비용**: K_max=3 반복 × 각 후보당 60개 태스크 실행이 필요하여 메타 진화 과정 자체의 연산/API 비용이 상당하며, 대규모·장기 태스크 환경에서는 비용이 급격히 증가할 수 있다.

- **단일 생존자 예산(K=1)**: 현재 설정에서 매 반복 1개 부모만 유지하므로 다양성 손실에 따른 조기 수렴 위험이 있으며, 더 큰 K에서의 탐색-활용 트레이드오프와 로컬 최적 탈출 능력은 미검증이다.

- **TaskCraft 의존 진화**: 메타 진화를 TaskCraft 한 곳에서만 수행하고 전이하는 방식이어서, 전이 대상 벤치마크의 고유 특성을 활용할 수 없고 도메인 특화 진화와 범용 진화 간 최적 균형은 탐구되지 않았다.

- **Manage 모듈 부재**: 다수의 진화된 시스템이 Manage 컴포넌트를 N/A로 남기며, 장기 운용 시 메모리 팽창·검색 품질 저하·구식 정보 유지 문제가 잠재한다.

- **평가 도메인 편향**: 주로 정보 검색·deep research 중심 벤치마크에 한정되어, 코딩, 수학 추론, 로봇 제어, 멀티모달 등 다른 에이전틱 도메인에서의 효과는 미검증이다.

- **LLM 설계자 의존성**: Design 단계가 LLM의 아키텍처 재설계 능력에 전적으로 의존하므로, 더 약한 백본에서는 창의적 변형 생성이 제한되어 진화 효과가 떨어질 가능성이 있다.

- **평가 메트릭의 3차원 축약**: (성능, 비용, 지연) 세 축으로 Pareto 정렬하는 방식이 강건성·안정성·프라이버시 같은 추가 품질 차원을 포착하지 못한다.

- **실제 영향**: 이 한계들은 MemEvolve가 현재 deep research류 태스크에는 즉시 적용 가능하지만, 실시간 로봇 제어나 저자원 디바이스 환경처럼 진화 예산이 작거나 태스크 다양성이 극단적인 상황에서는 추가 연구 없이는 그대로 이식하기 어렵다는 점을 시사한다.
