# MemEvolve: Meta-Evolution of Agent Memory Systems

> **논문 정보**: Guibin Zhang, Haotian Ren, Chong Zhan, Zhenhong Zhou, Junhao Wang, He Zhu, Wangchunshu Zhou, Shuicheng Yan (OPPO AI Agent Team, LV-NUS lab)
> **arXiv**: 2512.18746v1 (2025.12.21) | **학회**: -
> **코드**: https://github.com/bingreeky/MemEvolve

---

## Problem

기존 LLM 에이전트의 자기 개선(self-improving) 메모리 시스템은 **고정된 메모리 아키텍처** 위에서 경험만 축적하는 구조로, 메모리 인터페이스 자체는 한 번 설계되면 변하지 않는다. 이로 인해 특정 태스크 도메인에서는 효과적인 메모리 설계가 다른 도메인에서는 오히려 성능을 저하시키는 문제가 발생한다. 예를 들어, 궤적(trajectory)에서 재사용 가능한 API를 추출하는 메모리는 웹 브라우징에서는 유효하지만 수학적 추론에서는 거의 무용하며, 자기 비판(self-critique) 기반 메모리는 추론 집약적 태스크에서 강하지만 코딩/도구 사용 시나리오에서는 효과가 떨어진다. ExpeL은 단순한 QA/embodied 환경용으로 설계되어 장기 심층 연구 벤치마크에서 오히려 기준선보다 낮은 성능을 보인다. 즉, **메모리 아키텍처의 정적 특성이 다양한 태스크 환경에 대한 범용 자기 진화의 근본적 병목**이다.

---

## Motivation

인간 학습자에 비유하면, 단순히 경험을 축적하는 "숙련 학습자(skillful learner)"와 학습 전략 자체를 과목·상황에 따라 동적으로 조정하는 "적응 학습자(adaptive learner)"는 근본적으로 다르다. 문학 분석에서는 암기를, 수학에서는 풀이 템플릿 추상화를 우선하는 식으로 메타 전략을 바꾸는 것이 진정한 적응이다. 에이전트 메모리 시스템도 마찬가지로, **경험 축적(1차 진화)뿐 아니라 메모리 아키텍처 자체의 메타 진화(2차 진화)**를 동시에 수행해야 한다는 관찰이 핵심 직관이다. 이를 이중 수준 최적화(bilevel optimization)로 정형화하면, 내부 루프에서 고정된 메모리 시스템 아래 경험을 쌓고, 외부 루프에서 그 성능 피드백을 바탕으로 메모리 아키텍처를 진화시키는 선순환을 구축할 수 있다.

## Method

1. **모듈형 메모리 설계 공간**: 임의의 메모리 시스템 Ω를 네 가지 기능 모듈로 분해한다 — **Encode**(경험을 구조화된 표현으로 변환), **Store**(영속 메모리에 통합), **Retrieve**(태스크 관련 메모리 컨텍스트 검색), **Manage**(통합·망각·정리).

2. **EvolveLab 통합 코드베이스**: Voyager, ExpeL, Generative Agents, DILU, AWM, Mobile-E, Dynamic Cheatsheet, SkillWeaver, G-Memory, Agent-KB, Mem^p, EvolveR 등 12개 대표 메모리 시스템을 위 네 모듈 인터페이스(`BaseMemoryProvider`)로 재구현. 온라인/오프라인 평가 모드와 다수 벤치마크를 통합 지원.

3. **이중 진화 프로세스 (Dual-Evolution)**:
   - **내부 루프 (경험 진화)**: 각 후보 메모리 시스템 Ω_j^(k) 아래 에이전트가 60개 태스크 궤적을 실행하며 메모리를 빈 상태에서 축적. 각 궤적에서 성공률, 토큰 소비, 지연의 3차원 피드백 벡터를 수집하고 집계.
   - **외부 루프 (아키텍처 진화)**: 메타 진화 연산자 F가 후보 집합의 성능 요약을 기반으로 상위 K개 아키텍처를 선택하고, 각 부모에서 S개의 변형 후손을 생성.

4. **Pareto 기반 아키텍처 선택**: 성능·비용·지연의 3차원에 대해 비지배 정렬(non-dominated sorting)을 수행하여 Pareto 순위를 매기고, 동일 순위 내에서는 주요 성능 지표로 추가 정렬. 상위 K(=1)개를 부모로 선택.

5. **진단-설계 진화 (Diagnose-and-Design)**:
   - **진단 단계**: 부모 아키텍처의 실행 궤적을 리플레이하며 검색 실패, 비효율적 추상화, 저장 비효율 등의 구조화된 결함 프로파일을 생성.
   - **설계 단계**: 결함 프로파일을 조건으로 네 모듈의 구현만 변경하여 S(=3)개의 유효한 후손 변형을 생성. 각 변형은 인코딩 전략, 저장 규칙, 검색 제약, 관리 정책이 다르지만 통합 인터페이스 호환.

6. **반복 진화**: 기본 설정에서 K_max=3 반복 수행. 매 반복마다 40개 새 태스크 + 이전 20개 재사용으로 안정적 비교. 최종 진화된 메모리 시스템을 고정하여 미지의 벤치마크에 전이 평가.

## Key Contribution

1. **EvolveLab 통합 코드베이스**: 12개 대표 자기 개선 메모리 시스템을 Encode/Store/Retrieve/Manage의 4모듈 인터페이스로 재구현한 최초의 표준화된 실험 플랫폼. 기존에 산재했던 메모리 시스템들의 체계적 비교를 가능하게 함.

2. **메모리 아키텍처의 메타 진화 프레임워크**: 경험 축적(1차 진화)과 메모리 아키텍처 자체의 진화(2차 진화)를 이중 루프로 결합한 최초의 프레임워크. 고정된 메모리 아키텍처의 한계를 구조적으로 해결.

3. **진단-설계 진화 연산자**: Pareto 기반 다목적 선택과 궤적 리플레이 진단을 결합하여, 성능·비용·지연의 균형을 유지하면서 모듈 단위로 제어 가능한 아키텍처 진화를 실현.

4. **교차 태스크/모델/프레임워크 일반화 검증**: TaskCraft에서 진화한 메모리를 WebWalkerQA, xBench, GAIA 및 이종 에이전트 프레임워크(CK-Pro, OWL)에 무수정 전이하여 일관된 성능 향상을 실증.

## Experiment & Results

- **벤치마크**: GAIA, WebWalkerQA, xBench-DeepSearch, TaskCraft 4종. 메타 진화는 TaskCraft에서만 수행하고 나머지 3개에는 전이 평가.
- **에이전트 프레임워크**: SmolAgent(경량 2-에이전트), Flash-Searcher(고성능 단일 에이전트), CK-Pro(3-에이전트), OWL(계층적 멀티 에이전트).
- **주요 수치**:
  - Flash-Searcher+MemEvolve pass@1: GAIA **73.33%** (No-Memory 69.09%), xBench **74.0%** (69.0%), WebWalkerQA **74.71%** (71.18%) — 전 벤치마크에서 **3.54~5.0%** 안정적 향상.
  - Flash-Searcher+MemEvolve pass@3: GAIA **80.61%**, xBench **78.0%**, WebWalkerQA **81.18%** — OWL-Workforce, CK-Pro 등 강력한 멀티 에이전트 시스템을 상회.
  - SmolAgent+MemEvolve: xBench pass@1 51%→57% (+6%), pass@3 68%.
  - **교차 LLM 일반화**: Kimi K2에서 WebWalkerQA **17.06%** 향상, TaskCraft **10.0%** 향상 (GPT-5-Mini로 진화한 메모리를 무수정 전이).
  - **교차 프레임워크 일반화**: TaskCraft에서 진화한 메모리를 CK-Pro, OWL에 직접 전이 시 일관된 성능 향상.
  - **비용 효율**: API 비용은 No-Memory 기준선과 유사 (GAIA: $0.085 vs $0.086, xBench: $0.136 vs $0.141).
- **기존 메모리 시스템 비교**: 7개 기존 시스템(Voyager, ExpeL, Generative, DILU, AWM, Mobile-E, Cheatsheet) 중 어느 것도 3개 벤치마크 전체에서 일관된 향상을 달성하지 못함. ExpeL은 전 벤치마크에서 기준선 이하, DILU는 GAIA에서 -2.42%.
- **Ablation**: 진화 과정에서 초기에는 공격적 변형과 보수적 변형이 경쟁하며, 3라운드에 걸쳐 텍스트 인사이트+재사용 도구+주기적 DB 유지보수를 갖춘 최종 아키텍처(Cerebra)로 수렴.

## Limitation

- **진화 비용**: 3회 반복 × 각 후보당 60개 태스크 실행이 필요하여 메타 진화 과정 자체의 연산/API 비용이 상당하며, 대규모 또는 지연이 긴 태스크 환경에서는 비용이 급격히 증가할 수 있다.
- **단일 생존자 예산(K=1)**: 현재 설정에서 매 반복 1개 부모만 유지하므로, 다양성 손실로 인한 조기 수렴 위험이 있으며, 더 큰 K 값에서의 탐색-활용 트레이드오프는 미검증.
- **TaskCraft 의존 진화**: 메타 진화를 TaskCraft에서만 수행하고 전이하는 방식으로, 전이 대상 벤치마크의 고유 특성을 활용할 수 없다. 도메인 특화 진화와 범용 진화 간 최적 균형은 탐구되지 않았다.
- **메모리 관리(Manage) 모듈 부재**: 다수의 진화된 시스템이 Manage 컴포넌트를 N/A로 남기며, 장기 운용 시 메모리 팽창과 검색 품질 저하 문제가 잠재한다.
- **평가 범위**: 주로 정보 검색·연구 중심 벤치마크에 한정되며, 코딩, 수학 추론, 로봇 제어 등 다른 에이전틱 도메인에서의 효과는 미검증이다.
