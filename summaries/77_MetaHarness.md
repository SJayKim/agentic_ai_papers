# Meta-Harness: End-to-End Optimization of Model Harnesses

> **논문 정보**: Yoonho Lee, Roshen Nair, Qizheng Zhang (Stanford), Kangwook Lee (KRAFTON), Omar Khattab (MIT), Chelsea Finn (Stanford)
> **arXiv**: 2603.28052 (2026.03)
> **코드**: https://github.com/stanford-iris-lab/meta-harness-tbench2-artifact

---

## Problem
- LLM 시스템의 성능은 모델 가중치뿐 아니라 "하네스(harness)"—모델에 어떤 정보를 저장·검색·제시할지 결정하는 코드—에 크게 의존한다.
- 같은 벤치마크에서도 하네스 변경만으로 최대 6배의 성능 격차가 발생할 수 있다.
- 그러나 하네스 엔지니어링은 여전히 대부분 수작업으로 이루어지며, 실패 사례를 수동 점검하고 휴리스틱을 반복 조정하는 방식에 의존한다.
- 기존 텍스트 최적화 기법(OPRO, TextGrad, AlphaEvolve 등)은 하네스 최적화에 적합하지 않다.
- 이들은 피드백을 과도하게 압축하여 스칼라 점수만 참조하거나, 짧은 템플릿에 국한된 정보만 활용하기 때문이다.
- 하네스는 장기적 행동에 영향을 미치므로, 한 번의 저장·검색·제시 결정이 여러 추론 단계 뒤에야 효과를 나타내는 특성이 있다.
- 기존 방법들의 반복 당 활용 가능한 컨텍스트는 100~30,000 토큰에 불과하여, 하네스 탐색에 필요한 진단 정보량에 크게 못 미친다.
- 따라서 전체 실행 이력에 선택적으로 접근하여 하네스를 자동 최적화하는 새로운 접근법이 필요하다.

---

## Motivation
- 기존 텍스트 최적화 방법들은 피드백 정보를 요약하거나 스칼라 점수로 압축하여 전달하는 "정보 손실" 문제를 공통적으로 가진다.
- 하네스 최적화에서는 실행 트레이스(프롬프트, 도구 호출, 모델 출력, 상태 업데이트)까지 포함한 풍부한 진단 정보가 필요하다.
- 한 번의 평가가 최대 10,000,000 토큰의 진단 정보를 생성할 수 있어, 기존 방법의 피드백 예산보다 약 3자릿수 이상 크다.
- 코딩 에이전트의 최근 발전으로, 에이전트가 파일시스템을 통해 대규모 이력을 선택적으로 탐색하는 워크플로우가 비로소 실용적이 되었다.
- 검색 증강 생성(RAG), 메모리 기반 에이전트 등 선행 연구는 외부 컨텍스트에 적응적으로 접근하는 것이 유용함을 보여주었다.
- 이 아이디어를 하네스 최적화에 적용하면, 제안자(proposer)가 grep, cat 같은 표준 파일 연산으로 과거 코드와 실행 로그를 직접 탐색할 수 있다.
- 이를 통해 "왜 실패했는가"에 대한 인과적 가설 수립이 가능해지며, 단순 점수 기반 최적화보다 훨씬 효과적인 탐색이 가능하다.
- Rich Sutton의 "The Bitter Lesson"과 일맥상통하게, 검색 공간이 접근 가능해지면 범용 에이전트가 수작업 솔루션을 능가할 수 있다.

---

## Method
1. **하네스 정의**: 하네스는 고정된 LLM을 감싸는 상태 보유 프로그램으로, 각 단계에서 모델이 보는 컨텍스트를 결정한다. 최적화 목표는 기대 보상을 최대화하는 하네스 H*를 찾는 것이다.

2. **외부 루프(Outer Loop) 구조**: Meta-Harness는 제안(Propose) → 평가(Evaluate) → 저장(Store)의 반복 루프로 동작한다.

3. **파일시스템 기반 이력 관리**: 평가된 모든 하네스의 소스 코드, 점수, 실행 트레이스를 디렉토리 단위로 파일시스템 D에 저장한다. 이 파일시스템이 제안자의 유일한 피드백 채널이 된다.

4. **에이전틱 제안자(Agentic Proposer)**: 제안자는 단순 LLM이 아닌 코딩 에이전트(Claude Code + Opus-4.6)로, grep/cat 등의 터미널 도구를 통해 파일시스템을 능동적으로 탐색한다.

5. **선택적 진단**: 파일시스템이 컨텍스트 윈도우보다 훨씬 크므로, 제안자가 어떤 정보를 검사할지 스스로 결정한다. 가장 까다로운 설정에서 반복당 중앙값 82개 파일을 읽고, 20개 이상의 이전 후보를 참조한다.

6. **파레토 프론티어 유지**: 정확도와 컨텍스트 비용 등 복수 목표에 대해 파레토 우위 기반으로 후보를 관리하되, 부모 선택 규칙은 부과하지 않는다.

7. **코드 공간 탐색의 이점**: 하네스를 프로그램으로 표현함으로써 코딩 모델이 일관된 알고리즘을 제안하는 자연스러운 정규화 편향이 생긴다. 전체 구조 변경이 가능하다.

8. **인터페이스 검증**: 제안된 하네스는 인터페이스 검증을 통과해야만 평가 단계에 진입한다.

9. **실행 규모**: 일반적인 실행에서 약 20회 반복에 걸쳐 60개 하네스를 평가한다. 텍스트 분류에서는 40개, 수학 추론에서는 109개 후보 하네스를 생성한다.

10. **최소 구조 원칙**: 고정 스캐폴드, 아카이브, 지속적 메모리 메커니즘 대신 제안자에게 제한 없는 파일시스템 접근을 부여하는 의도적으로 최소화된 설계이다.

11. **실행 트레이스 기반 인과 추론**: 제안자는 실행 트레이스를 검토하여 실패 원인을 추적하고, 혼동된 편집을 분리하며, 반복된 퇴행 후 안전한 수정으로 전환하는 행동을 보인다.

## Key Contribution
1. **하네스 최적화 자동화 프레임워크 제안**: 코딩 에이전트가 파일시스템을 통해 전체 실행 이력에 선택적으로 접근하여 하네스를 자동 최적화하는 Meta-Harness를 제안하였다.
2. **기존 텍스트 최적화 대비 10배 빠른 수렴**: 텍스트 분류에서 OpenEvolve, TTT-Discover 대비 1/10의 평가 횟수로 동일 성능에 도달하고, 최종 정확도는 10포인트 이상 초과하였다.
3. **실행 트레이스 접근의 핵심성 입증**: 점수만 제공 vs. 요약 추가 vs. 전체 트레이스 접근의 3단계 소거 실험을 통해, 원시 실행 트레이스 접근이 핵심 요소임을 실증하였다.
4. **다중 도메인 검증**: 온라인 텍스트 분류, 검색 증강 수학 추론, 에이전틱 코딩의 3개 도메인에서 일관된 성능 향상을 확인하였다.
5. **발견된 하네스의 일반화 능력**: 탐색에 사용되지 않은 9개 OOD 데이터셋과 5개 미경험 모델에서도 성능 향상이 전이됨을 보였다.
6. **수작업 에이전트 능가**: TerminalBench-2에서 자동 탐색으로 발견된 하네스가 수작업 엔지니어링 최고 수준의 에이전트를 초과하였다.

## Experiment & Results
- **온라인 텍스트 분류**: Meta-Harness는 ACE 대비 7.7포인트, MCE 대비 8.6포인트 향상된 **48.6%** 정확도를 달성하면서, 컨텍스트 토큰은 **11.4K**로 ACE(50.8K) 대비 4배 이상 적게 사용.
- **텍스트 최적화 비교**: Meta-Harness(중앙값 50.0, 최고 56.7) vs. OpenEvolve(39.1, 43.3) vs. TTT-Discover(34.1, 45.6) vs. GEPA(32.6, 40.2)로 크게 우위.
- **소거 실험**: 점수만 제공 시 최고 41.3, 요약 추가 시 38.7, **전체 트레이스 접근 시 56.7**로 급상승. 중앙값 기준으로도 전체 인터페이스(50.0)가 소거 조건의 최고치(41.3)를 초과.
- **OOD 일반화**: 탐색에 사용되지 않은 9개 데이터셋에서 평균 **73.1%** 정확도로 ACE(70.2%)를 2.9포인트 상회하고, 9개 중 6개 데이터셋에서 최고 성능.
- **수학 추론**: 200개 IMO 수준 문제에서 검색 미사용 대비 평균 **4.7포인트** 향상(38.8% vs. 34.1%). 5개 미경험 모델 전체에서 일관된 개선.
- **TerminalBench-2**: Opus 4.6 기반 **76.4%** 통과율로 Terminus-KIRA(74.7%)를 상회하여 리더보드 2위. Haiku 4.5 기반에서는 **37.6%**로 전체 1위(Goose 35.5% 대비 2.1포인트 우위).
- **탐색 효율성**: 텍스트 분류에서 Meta-Harness는 **4회 평가**만으로 다른 텍스트 최적화 방법의 60회 평가 후 최종 성능에 매칭.

---

## Limitation
- **제안자 의존성**: 실험이 단일 코딩 에이전트(Claude Code + Opus-4.6)에 국한되어, 다른 제안자 에이전트에서의 효과 변화에 대한 체계적 연구가 부족하다.
- **계산 비용**: 반복당 최대 10,000,000 토큰의 진단 정보를 생성하며, 제안자가 반복마다 82개 파일을 읽는 등 상당한 계산 자원이 필요하다.
- **TerminalBench-2 평가 설계**: 탐색과 최종 평가를 동일한 89개 태스크에서 수행하여, 별도의 탐색/테스트 분할이 없다.
- **코드 공간 탐색의 해석 가능성 한계**: 복잡한 하네스 코드의 동작을 완전히 이해하기 어려울 수 있다.
- **모델-하네스 공동 최적화 미탐구**: 하네스만 최적화하고 모델 가중치는 고정하므로, 하네스와 모델을 함께 진화시키는 잠재적 시너지를 활용하지 못한다.
- **2026년 초 코딩 에이전트 능력에 의존**: 이 워크플로우는 최근의 코딩 에이전트 발전에 의해 비로소 가능해진 것으로, 기술적 시의성에 강하게 결합되어 있다.
