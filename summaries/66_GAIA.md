# GAIA: A Benchmark for General AI Assistants

> **논문 정보**: Grégoire Mialon, Clémentine Fourrier, Craig Swift, Thomas Wolf, Yann LeCun, Thomas Scialom (FAIR Meta, HuggingFace, AutoGPT, GenAI Meta)
> **arXiv**: 2311.12983 (2023.11) | **학회**: ICLR 2024
> **코드**: https://huggingface.co/gaia-benchmark

---

## Problem

대규모 언어 모델(LLM)이 빠르게 발전하면서 기존 벤치마크들은 해를 거듭할수록 더 빠른 속도로 포화(saturation)되고 있다.
MMLU, GSM8k 같은 전문 지식 기반 벤치마크는 이미 GPT-4가 인간 전문가 수준(MMLU 86.4%)에 근접하며, 데이터 오염(data contamination) 가능성까지 더해져 진정한 일반화 능력을 측정하지 못하고 있다.
현재의 벤치마크 설계 트렌드는 "인간에게 점점 더 어려운 과제"를 추구하지만, 인간에게 어려운 과제가 반드시 LLM에게도 어려운 것은 아니라는 역설이 존재한다.
반대로, 웹 브라우징·멀티모달 처리·코드 실행·다단계 추론이 동시에 필요한 실세계 복합 태스크에서는 인간이 92% 정답률을 보이는 반면 GPT-4는 플러그인을 갖춰도 15%에 그친다.
객관식 포맷(MMLU 등)은 잘못된 추론 과정으로도 정답을 찍을 수 있어 오염 평가를 어렵게 하며, 열린 생성(open-ended generation) 평가는 인간·모델 기반 심사에 의존해 확장성과 재현성이 떨어진다.
결국 일반 목적 AI 어시스턴트(General AI Assistant)의 실제 능력을 측정할 수 있는, 비-게임어빌리티(non-gameability)를 갖춘 자동 채점형 벤치마크가 부재한 상황이다.

---

## Motivation

저자들은 Proof of Work 알고리즘에서 영감을 얻어, "개념적으로는 단순하지만 정확한 긴 행동 시퀀스 실행을 요구하며, 결과는 쉽게 검증 가능한" 태스크가 AI 시스템 평가의 이상적인 형태라고 제안한다.
AI 어시스턴트 태스크는 본질적으로 다양하고 불확실한 실세계 접근이 필요하므로 이러한 조건을 자연스럽게 충족한다.
Morris et al.(2023)의 Levels of AGI 프레임워크에서, GAIA 같은 실세계 복합 태스크를 해결할 수 있는 시스템은 ChatGPT 수준보다 한 단계 높은 "Competent General AI"에 해당한다.
또한 t-AGI 개념(인간이 특정 시간 안에 수행하는 작업을 AI가 동일 시간 내에 수행)에 비추어, GAIA 문제는 인간이 6~17분 소요하는 범위에 위치하여 정량적 기준점을 제공한다.
도구 증강 LLM(tool-augmented LLM)이 실제로 어떤 격차를 메우고 어떤 부분에서 실패하는지를 실증적으로 드러낼 필요가 있다.
벤치마크가 실세계에 뿌리박힌 만큼 시간이 지나면 "decay(노후화)"하므로, 정적 벤치마크가 아닌 지속적으로 갱신되는 동적 벤치마크의 방법론적 기반을 구축하는 것도 중요한 동기이다.

## Method

1. **4가지 설계 원칙 확립**: (a) 실세계성 — 정답이 사전학습 데이터의 평문으로 존재하지 않도록 설계, (b) 해석 가능성 — 비전문가도 92% 맞힐 수 있는 개념적 단순성, (c) 비-게임어빌리티 — 다단계 완료 필수 + 개방형 단답으로 우연 정답 불가, (d) 사용 편의성 — 팩토이드 단답으로 자동 채점.
2. **질문 포맷 설계**: 텍스트 기반 프롬프트 + 선택적 첨부 파일(이미지, 스프레드시트, 오디오, 비디오, PDF 등). 답은 문자열 한두 단어, 숫자, 또는 쉼표 구분 리스트 중 하나로 제한되어 단일 정답을 보장.
3. **질문 생성 프로세스**: 저자가 초기 시드 질문을 설계하고, 어노테이터에게 예시와 가이드라인(Appendix D)을 제공하여 추가 생성. 질문은 Wikipedia·Papers With Code·arXiv 같은 신뢰할 수 있는 "sources of truth"를 기반으로 작성하며, 출처 버전을 명시하여 노후화를 방지.
4. **질문 유형 3종**: (i) 웹 기반 정보 결합형, (ii) 첨부 파일 자체 완결형, (iii) 자기완결 퍼즐형. 대부분은 여러 출처에서 정보를 찾아 결합·변환하여 단일 답을 도출.
5. **어노테이션 메타데이터**: 질문 생성자가 답 외에 사용한 도구 목록, 수행 단계 수, 소요 시간을 함께 기록하여 난이도 분류의 근거를 마련.
6. **독립 2인 교차 검증**: 원 생성자 + 2명의 새 어노테이터가 독립적으로 답하여 3명 일치 시 승인. 68%는 수정 없이 통과, 나머지는 단순 수정 또는 제거. 질문당 평균 ~2시간 소요.
7. **3단계 난이도 분류**: Level 1 — 도구 0~1개, 5단계 이하. Level 2 — 5~10단계, 여러 도구 결합. Level 3 — 임의 길이 행동 연쇄, 임의 도구 조합. 단계/도구 수는 엄격한 경계가 아니며 복잡한 웹 네비게이션 등 정성적 요소도 반영.
8. **능력 커버리지**: 웹 브라우징(~355), 멀티모달(~154), 코딩(~138), 다양한 파일 포맷 읽기(~129), 기타(~32). 단일 질문이 여러 능력을 동시에 요구하는 경우가 많음.
9. **평가 방식**: 준-정확 매칭(quasi exact match) — 정답 타입에 따라 정규화 후 문자열 비교. 숫자는 쉼표·단위 제거, 문자열은 관사·축약어 제거 등. 시스템 프롬프트로 "FINAL ANSWER: …" 템플릿 강제.
10. **3회 평균 실행**: API 접근 가능 모델은 3회 실행 후 평균 점수를 보고하여 토큰 생성 무작위성 완화.
11. **데이터셋 분할**: 총 466개 중 Dev 166개(답 포함 공개) + Test 300개(답 비공개, HuggingFace 리더보드에서 자동 채점).
12. **베이스라인 비교군**: GPT-4, GPT-4 Turbo, GPT-4 + 플러그인(수동 선택), AutoGPT-4(자동 도구 선택), 웹 검색엔진, 인간 어노테이터.
13. **플러그인 선택 전략**: 플러그인은 API가 없어 ChatGPT 수동 조회. 태스크마다 "링크 리더 + 웹 브라우저 + 계산 도구" 조합을 최적 추정. 이로 인해 GPT-4+플러그인 점수는 "oracle" 추정치로 간주.
14. **재현성 보완**: 정답이 단일 사실이므로 최종 답만 비교하여 모델 출력 변동성에 강건. 단 플러그인 조합은 시간에 따라 변하므로 완전 재현은 어려움을 명시.

## Key Contribution

1. **AGI 진전의 정량적 기준점 제시**: 인간 92% vs GPT-4+플러그인 15%의 격차를 통해, 현재 LLM이 전문 시험은 통과해도 일상적 복합 태스크에서는 크게 뒤처진다는 점을 드러내어 AGI까지의 남은 간극을 측정 가능한 수치로 표현.
2. **비-게임어빌리티 방법론**: 개방형 단답 + 다단계 완료 필수 + 사전학습 데이터 평문 부재 + 출처 버전 명시를 결합하여 데이터 오염과 우연 정답을 원천 차단하는 벤치마크 설계 원칙을 정립.
3. **실세계-뿌리 벤치마크 패러다임**: 폐쇄 환경(AgentBench 등)이 아닌 열린 웹과 실제 파일을 사용하여, 특정 API 학습이 아닌 일반 능력을 측정. 이는 이후 OWL, WORKFORCE 등 에이전트 연구의 표준 평가 기준이 됨.
4. **도구 증강의 실증적 검증**: GPT-4 단독 vs GPT-4+플러그인의 점수 격차를 통해 tool-augmented LLM이 핵심 연구 방향임을 정량적으로 확인.
5. **자율 오케스트레이션의 한계 노출**: AutoGPT-4가 자동 도구 선택 능력에도 불구하고 GPT-4 단독보다 낮은 점수를 기록하여, 현재의 자율 에이전트 프레임워크가 프롬프트 설계·생성 파라미터 측면에서 미성숙함을 드러냄.
6. **재현 가능한 자동 채점 파이프라인**: 준-정확 매칭 스코어 함수와 HuggingFace 리더보드를 공개하여 후속 연구가 동일 프로토콜로 비교 가능한 인프라 제공.
7. **질문 생성 방법론 공개**: 이중 교차 검증, 메타데이터 어노테이션, 난이도 분류 기준 등을 문서화하여 커뮤니티가 GAIA를 확장하거나 유사 벤치마크를 구축할 수 있도록 지원.

## Experiment

총 466개 질문(Level 1: 쉬움, Level 2: 중간, Level 3: 어려움)에 대해 5종 시스템을 3회씩 실행하여 평균 정답률을 측정.
**인간 어노테이터**: 전체 평균 92% 정답률, Level별로 거의 동일한 성능. Level 1 평균 소요 시간 약 6분, Level 3 약 17분.
**GPT-4 + 플러그인(수동 선택, oracle)**: Level 1 ~30%, Level 2 ~9.7%, Level 3 ~0%, 전체 ~15%. 백트래킹·쿼리 재조정·장기 계획 실행 등 고급 행동을 보였으나 복잡한 다단계 태스크에서는 실패.
**GPT-4 Turbo (도구 없음)**: Level 1에서 ~9~10%대, Level 2·3는 거의 0%에 수렴.
**GPT-4 (플러그인 없음)**: Level 1 ~10%, Level 2 ~5%, Level 3 0%. 멀티모달·파일 처리 능력 부재가 뚜렷.
**AutoGPT-4**: Level 1 ~5% 수준으로 GPT-4 단독보다도 낮음, Level 2·3은 거의 0%. 자동 도구 선택에도 불구하고 프롬프트 엔지니어링 한계로 성능 저조, 실행 속도도 타 모델 대비 매우 느림.
**웹 검색엔진 베이스라인**: Level 1 일부 질문에서 첫 페이지 결과로 답 추출 가능하나 Level 2 이상에서는 무력, 인간 사용자가 결과를 스킴해야 해 LLM 어시스턴트보다 느림.
**능력별 Level 1 분석**: GPT-4+플러그인은 웹 브라우징·코딩·파일 처리에서 최대 ~20%, 멀티모달은 여전히 취약. 도구 없는 GPT-4는 "웹 브라우징" 태스크에서도 부분 점수를 얻는데 이는 답을 암기했거나 중간 정보가 사전학습에 포함된 경우.
**질문 검증 통계**: 466개 중 68%가 원본 그대로 통과, 32%는 수정 또는 제거. 어노테이터 정답률 92%가 "인간 점수"로 기록됨.
**Level 3의 절벽**: 모든 AI 변형이 Level 3에서 ~0%를 기록하여, 10단계 이상 복합 도구 사용이 현재 LLM의 절대적 한계임을 시각화.
**응답 시간 트레이드오프**: Figure 4에서 GPT-4+플러그인이 점수 대비 시간 효율에서 가장 우수, 인간은 정확도 최고지만 시간 소요 큼.

## Limitation

1. **추론 과정(trace) 미평가**: 현 버전은 최종 답만 비교하므로 잘못된 과정으로 맞힌 경우나 우수한 추론 과정을 평가하지 못함. 인간/모델 기반 trace 평가는 향후 과제로 남김.
2. **규모 한계**: 466개는 MMLU(~15,000)에 비해 작아 통계적 세분 분석이 제한적. 다만 저자들은 단일 정답 개방형 질문의 질을 확보하기 위해 양보다 질을 우선했다고 명시.
3. **언어·문화적 편향**: 전 질문이 표준 영어로만 작성되고 주로 영어 웹 페이지에 의존. 전 세계 인구 80%가 비영어 사용자이며 웹 콘텐츠 절반이 비영어인 현실을 반영하지 못해 AI 어시스턴트의 범용성 증명으로는 불충분.
4. **웹 링크 노후화 위험**: Wikipedia 문서 업데이트·플러그인 소멸·웹 페이지 변경으로 질문이 시간이 지나면 풀 수 없게 될 수 있음. 버전 명시로 완화하지만 완전한 해결책은 아님.
5. **플러그인 재현성 부재**: ChatGPT 플러그인은 API가 없고 스토어에서 자주 변경·제거되어 GPT-4+플러그인 점수는 "oracle" 추정치일 뿐 엄밀한 재현이 불가능. 공식 검색 도구도 유료 콘텐츠 우회 문제로 일시 제거된 전례가 있음.
6. **모호성 제거 비용**: 단일 정답을 보장하기 위해 질문을 과도하게 구체화(세부 조건·쉼표 처리·버전 명시 등)해야 해, 실제 사용자가 덜 구체적으로 질문하는 상황과 괴리. 실세계에서는 출처를 인용하거나 여러 가능성을 제시하는 어시스턴트가 유용하지만 GAIA는 이를 평가하지 못함.
7. **웹 행동 제한**: 클릭 이외의 행동(파일 업로드, 댓글 작성, 예약 등)은 웹사이트 스팸 우려로 배제되어, 실세계 어시스턴트가 수행할 쓰기 행동 평가 불가.
8. **세부 도구 사용 분석 부재**: 도구 호출 로그가 OpenAI API에서 제공되지 않아 모델이 어떤 도구를 어떤 순서로 호출했는지 정밀 분석이 불가능. 오픈소스 도구 증강 LLM의 추가 평가가 필요.
