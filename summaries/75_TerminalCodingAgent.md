# Building Effective AI Coding Agents for the Terminal: Scaffolding, Harness, Context Engineering, and Lessons Learned

> **논문 정보**: Nghi D. Q. Bui (OpenDev)
> **arXiv**: 2603.05344 (2026.03)
> **코드**: https://github.com/opendev-to/opendev

---

## Problem
- 기존 AI 코딩 어시스턴트는 IDE 플러그인 형태로 설계되어, 소스 관리·빌드·배포가 이루어지는 터미널 환경에서의 자율적 장시간 개발 작업에 적합하지 않다.
- Terminal-Bench에서 최첨단 모델조차 CLI 작업의 65% 미만만 해결하고, LongCLI-Bench에서는 장기 작업 통과율이 20% 이하로 나타나, 장시간 터미널 운영의 근본적 한계가 드러났다.
- 유한한 컨텍스트 윈도우 관리 문제: 도구 출력(파일 내용, 명령 결과, 검색 결과)이 일반 세션에서 컨텍스트의 70~80%를 차지하여, 에이전트의 추론 능력이 급격히 저하된다.
- 에이전트가 임의 셸 명령을 실행할 수 있어 파괴적 작업(rm -rf 등)을 방지할 안전 메커니즘이 필요하나, 기존 단일 안전 레이어로는 부족하다.
- 시스템 프롬프트의 영향력이 대화가 길어지면서 쇠퇴(attention decay)하여, 30회 이상 도구 호출 후 에이전트가 지시를 위반하는 현상이 반복적으로 관찰된다.
- 외부 도구 확장 시 모든 MCP 스키마를 로드하면 첫 메시지 전에 컨텍스트의 40%를 소비하는 비효율이 발생한다.
- 기존 오픈소스 프레임워크는 벤치마크 지향이거나 기술 보고서가 없고, Claude Code 등 상용 시스템은 비공개여서 설계 결정을 학습할 수 없다.

---

## Motivation
- 터미널은 소프트웨어 개발의 운영 중심부로, 소스 관리·빌드 시스템·원격 SSH·헤드리스 서버 환경을 네이티브로 지원하므로 CLI 기반 에이전트가 IDE 통합 도구보다 더 높은 자율성을 제공할 수 있다.
- Compound AI System 패러다임에 기반하여, 단일 모노리식 LLM이 아닌 다중 에이전트·워크플로우의 구조적 앙상블로 설계하면 비용·지연·능력 간 트레이드오프를 워크플로우별로 최적화할 수 있다.
- 컨텍스트를 단순 버퍼가 아닌 "예산(budget)"으로 취급하고, 점진적 감축 전략을 적용하면 긴급 압축 없이도 세션 수명을 크게 연장할 수 있다는 통찰이 핵심이다.
- 사고(thinking)와 행동(action)을 분리하면 LLM이 도구 스키마의 압박 없이 깊이 있는 추론을 수행하며, 이는 "신중하게 생각하라"는 프롬프트보다 구조적으로 더 효과적이다.
- 안전 제약을 런타임 권한 검사 대신 스키마 수준에서 적용(위험 도구를 아예 보이지 않게 제거)하면, 모델이 우회를 시도할 수 없어 근본적으로 더 견고하다.
- 이 논문의 목적은 새로운 알고리즘적 돌파가 아니라, 생산 수준의 에이전트 코딩 시스템 구축에서 얻은 설계 결정·트레이드오프·교훈을 체계적으로 공유하는 것이다.
- 오픈소스 터미널 네이티브 대화형 코딩 에이전트에 대한 최초의 포괄적 기술 보고서로서, 폐쇄적 산업 관행과 학술 담론 사이의 격차를 해소하고자 한다.

## Method
1. **4계층 시스템 아키텍처**: Entry & UI → Agent → Tool & Context → Persistence의 계층 구조로 설계.
   - Entry 계층: CLI, TUI(Textual 기반), Web UI(FastAPI+WebSocket) 3가지 프런트엔드를 공유 UICallback 계약으로 통합.
   - Persistence 계층: JSON/JSONL 기반 세션 저장, 설정 관리(4단계 계층: built-in → 환경변수 → 사용자 글로벌 → 프로젝트 로컬), shadow git 스냅샷 undo 시스템.

2. **Compound AI System 기반 다중 모델 아키텍처**: 5가지 모델 역할(Action, Thinking, Critique, Vision, Compact)을 워크플로우별로 독립 바인딩.
   - 각 모델은 지연 초기화(lazy initialization)로 세션에서 실제 사용 시에만 생성.

3. **확장 ReAct 실행 루프(4단계)**: Phase 0(단계적 컨텍스트 관리) → Phase 1(도구 없는 사고) → Phase 2(도구 포함 행동) → Phase 3(결정·디스패치·둠루프 탐지).
   - 사고 단계: OFF/LOW/MEDIUM/HIGH 4단계 깊이, HIGH에서는 자기 비평(self-critique) 자동 포함.
   - 둠루프 탐지: 최근 20회 도구 호출의 MD5 핑거프린트를 추적, 동일 호출 3회 반복 시 경고 → 재발 시 승인 기반 정지로 2단계 에스컬레이션.

4. **이중 모드 운영(Plan/Normal)**: Plan Mode는 Planner 서브에이전트에 위임(읽기 전용 도구만), Normal Mode는 전체 도구 접근.
   - 8종 서브에이전트(Code Explorer, Planner, PR-Reviewer, Security-Reviewer 등)를 스키마 필터링으로 격리, asyncio.gather()로 자동 병렬 실행.

5. **Adaptive Context Compaction (ACC)**: 5단계 점진적 압축 파이프라인.
   - 70%(경고) → 80%(관측 마스킹, ~15토큰/관측) → 85%(빠른 프루닝) → 90%(공격적 마스킹) → 99%(전체 LLM 요약).
   - 피크 관측 컨텍스트 소비를 약 54% 감소, 30턴 세션에서 긴급 압축 필요성을 자주 제거.

6. **이벤트 기반 System Reminder**: 8가지 이벤트 탐지기(도구 실패, 탐색 스파이럴, 미완료 todo 등)가 role:user 메시지로 적시 주입.
   - 리마인더별 가드레일 카운터(todo nudge 최대 2회, 에러 복구 최대 3회 등)로 과도한 반복 방지.

7. **도구 시스템**: 35개 내장 도구를 12개 핸들러 카테고리로 구성한 레지스트리 아키텍처.
   - edit_file: 9단계 체인-of-responsibility 퍼지 매칭.
   - MCP 도구 lazy discovery: search_tools로 키워드 검색 후 발견된 도구만 스키마에 포함, 기본 오버헤드를 40%에서 5% 미만으로 감소.
   - 5계층 방어-심층(defense-in-depth) 안전 아키텍처: 프롬프트 가드레일 → 스키마 도구 게이팅 → 런타임 승인 → 도구 검증 → 라이프사이클 훅.

8. **ACE(Agentic Context Engineering) 메모리 파이프라인**: 4단계(BulletSelector → Reflector → Curator → Playbook 적용)로 프로젝트별 경험을 플레이북 bullet로 축적.

## Key Contribution
1. **워크플로우별 LLM 구성 가능성**: Compound AI System 아키텍처로 5가지 인지 워크플로우를 독립 모델에 바인딩하여, 비용·지연·능력을 워크플로우별로 최적화.
2. **확장 ReAct 실행 파이프라인**: 표준 ReAct에 명시적 사고 단계와 선택적 자기 비평을 추가하고, 단계적 컨텍스트 압축을 추론 루프에 직접 통합.
3. **장기 세션 행동 조향**: 이벤트 기반 시스템 리마인더로 instruction fade-out에 대응하고, 조건부 프롬프트 합성으로 불필요한 지시 제거.
4. **토큰 효율적 확장성 및 방어-심층 안전**: MCP lazy discovery로 외부 도구 오버헤드를 최소화하고, 5계층 독립 안전 레이어로 단일 장애점 없는 보안 제공.
5. **컨텍스트 엔지니어링의 일급 관심사화**: ACC(5단계 점진적 압축), 이중 메모리, ACE 경험 메모리 파이프라인을 통합한 체계적 컨텍스트 관리.
6. **오픈소스 터미널 네이티브 코딩 에이전트의 최초 포괄적 기술 보고서**.

## Experiment & Results
- 이 논문은 아키텍처 설계·트레이드오프·교훈 공유에 초점을 맞추며, 정량적 수치는 주로 시스템 내부 측정에서 도출된다.
- **ACC 효과**: 피크 관측 컨텍스트 소비를 약 54% 감소시켜, 30턴 세션에서 긴급 압축 필요성을 대부분 제거.
- **도구 결과 최적화**: 단일 도구 호출의 컨텍스트 비용을 수천 토큰에서 100토큰 미만으로 감소. 8,000자 초과 출력은 스크래치 파일로 오프로드.
- **세션 수명 연장**: 도입 전 15~20턴에서 컨텍스트 오버플로 발생하던 것이, 도입 후 30~40턴까지 연장.
- **MCP lazy discovery**: 외부 도구 스키마의 기본 컨텍스트 오버헤드를 40%에서 5% 미만으로 감소.
- **프롬프트 캐싱**: 안정 부분이 전체 시스템 프롬프트의 80~90%를 차지하여, 캐시된 부분의 입력 토큰 비용을 약 88% 절감.
- **edit_file 퍼지 매칭**: 기존 2단계 매칭에서 9단계 체인으로 확장하여 "content not found" 에러의 대부분을 해결.
- **참고 벤치마크 수치**: Terminal-Bench 해결률 65% 미만, LongCLI-Bench 장기 작업 통과율 20% 미만, MCE 기법의 SWE-bench Verified에서 89.1% 해결률(기존 70.7% 대비).

## Limitation
- **체계적 정량 평가 부재**: SWE-bench 등 확립된 벤치마크에서의 정량적 성능 평가가 포함되지 않아, 기존 시스템 대비 우위를 검증할 수 없다.
- **고정 파라미터 한계**: 70% 압축 임계값, 3회 nudge 시도, 사고 깊이 6단계 등이 전역 고정값으로, 작업 복잡도에 따른 동적 적응이 없다.
- **메모리 파이프라인 확장성**: ACE 플레이북이 프로젝트 단위로만 운영되어, 프로젝트 간 지식 전이(cross-project transfer)가 불가능하다.
- **서브에이전트 협업 제한**: 서브에이전트 간 계층적 위임만 가능하고, 피어-투-피어 통신·공유 블랙보드 등 고급 협업 패턴이 미지원.
- **리마인더 수동 설계**: 24개 리마인더 템플릿과 주입 타이밍이 수동 엔지니어링이며, 강화학습 기반 자동 최적화가 적용되지 않았다.
- **CLI 전용 한계**: 터미널 네이티브 설계로 인해 IDE의 시각적 어포던스를 활용할 수 없다.
