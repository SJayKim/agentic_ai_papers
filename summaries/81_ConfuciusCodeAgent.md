# Confucius Code Agent: Scalable Agent Scaffolding for Real-World Codebases

> **논문 정보**: Sherman Wong, Zhenting Qi, Zhaodong Wang, Nathan Hu, Samuel Lin, Jun Ge, Erwin Gao, Wenlin Chen, Yilun Du, Minlan Yu, Ying Zhang (Meta, Harvard)
> **arXiv**: 2512.10398 (2025.12)
> **코드**: https://github.com/facebookresearch/cca-swebench

---

## Problem
- 현실 소프트웨어 엔지니어링 작업은 대규모 리포지토리에서 장기간 세션을 유지하며 복잡한 도구 체인을 안정적으로 조율하는 코딩 에이전트를 요구한다.
- 기존 연구용 코딩 에이전트(SWE-Agent 등)는 투명성은 높지만, 프로덕션 수준의 대규모 워크로드로 확장하면 성능이 크게 저하된다.
- 상용 코딩 에이전트는 실용적 성능은 강하지만, 확장성·해석가능성·제어가능성이 제한적이다.
- 대부분의 프레임워크가 flat한 대화 히스토리, 휴리스틱 프롬프트 엔지니어링, 또는 밀결합된 도구 파이프라인에 의존하여 장기 다단계 워크플로에 취약하다.
- 핵심 도전 과제 C1: 대규모 리포지토리에서 관련 코드를 효율적으로 탐지하고 분산된 모듈 간 다중 홉 추론을 수행하는 **Long-context reasoning** 문제가 존재한다.
- 핵심 도전 과제 C2: 세션 간 지속적 지식(재사용 패턴, 실패 모드 등)을 축적하는 **Long-term memory** 부재로 동일 실수를 반복하게 된다.
- 동일 백본 모델을 사용하더라도 스캐폴딩 전략에 따라 성능 격차가 매우 크게 나타나며, 에이전트 설계 자체가 핵심 연구 차원임을 시사한다.

---

## Motivation
- 에이전트 성능은 LLM 능력뿐 아니라 **에이전트 스캐폴딩**(오케스트레이션, 메모리 구조, 도구 추상화)에 크게 좌우된다는 경험적 관찰에 기반한다.
- 기존 프레임워크 대부분이 AX(Agent Experience), UX(User Experience), DX(Developer Experience)를 혼동하여 설계하는 문제를 지적한다.
- 예를 들어 사람용 로그를 모델 프롬프트에 그대로 넣으면 컨텍스트 팽창, 허위 앵커, 디버깅 어려움 등이 발생한다.
- AX/UX/DX를 독립적이면서 상호 보완적인 설계 원칙으로 분리하면, 확장 가능하고 분석 가능하며 재현 가능한 에이전트 행동의 기반을 마련할 수 있다.
- 에이전트의 도구 사용 행동을 사람이 수동으로 설계하는 것은 노동 집약적이고 도구 생태계 확장에 대응하지 못한다.
- 약한 모델 + 강한 스캐폴딩이 강한 모델 + 약한 스캐폴딩을 능가할 수 있다는 가설을 검증하고자 한다.
- 세션 간 지속적 학습을 통해 에이전트가 시간이 지날수록 효율적으로 개선되는 메커니즘이 필요하다.

---

## Method
1. **AX/UX/DX 3축 설계 철학**
   - AX: 에이전트의 인지 작업 공간으로, 모델이 수신하는 컨텍스트의 구조화와 도구 호출 방식을 관리한다.
   - UX: 인간 사용자를 위한 투명한 로그, 트레이스, 아티팩트 프리뷰를 제공한다.
   - DX: 개발자가 프롬프트, 도구, 메모리를 관찰·제어할 수 있는 가시성을 보장한다.
   - 사용자에게는 풍부한 diff 포함 스트리밍 업데이트를, 에이전트에는 압축된 결과 요약만을 제공하여 채널을 분리한다.

2. **계층적 작업 메모리 + 적응적 컨텍스트 압축 (C1 해결)**
   - 파일 시스템 기반 트리 구조 메모리(세션/엔트리/실행 단위 스코프)로 마크다운 문서와 메타데이터 태그를 저장한다.
   - 컨텍스트가 임계값에 도달하면 **Architect Agent**가 별도 호출로 대화 이력을 분석하여 구조화된 요약을 생성한다.
   - 요약이 이전 이력을 대체하되, 최근 메시지는 원본 그대로 유지하는 롤링 윈도우 방식을 적용한다.

3. **노트 테이킹 시스템 (C2 해결)**
   - 전용 note-taking agent가 세션 트라젝토리를 비동기적으로 분석하여 구조화된 마크다운 노트로 증류한다.
   - **Hindsight notes**로 실패 사례(컴파일 오류, 런타임 예외, 비생산적 전략)와 해결 방법을 기록한다.
   - 향후 세션에서 유사 실패 발생 시 기존 노트를 검색하여 즉시 해결책을 제시한다.

4. **Extension 시스템**
   - 모듈형 콜백 기반 확장으로, 출력 파싱·도구 호출·부작용 관리를 오케스트레이터에서 분리한다.

5. **Confucius Orchestrator**
   - LLM 호출 → 출력 파싱 → Extension 라우팅 → 메모리 업데이트의 반복 루프를 실행한다.
   - 네이티브 tool-use와 XML 태그 파싱 두 가지 인터페이스를 지원하여 모델 호환성을 확보한다.

6. **Meta-Agent: Build-Test-Improve 루프**
   - 고수준 자연어 명세로부터 에이전트 구성(프롬프트, Extension 배선, 메모리 정책)을 자동 생성한다.
   - 회귀 테스트 스위트에서 후보 에이전트를 실행하고, 실패 패턴을 관찰하여 프롬프트와 도구 래퍼를 자동 수정한다.

## Key Contribution
1. 대규모 코드베이스용 코딩 에이전트 CCA를 제안하며, AX/UX/DX 3축 설계 철학에 기반한 Confucius SDK 플랫폼을 공개하였다.
2. 계층적 작업 메모리와 Architect Agent 기반 적응적 컨텍스트 압축으로 장기 추론을 안정화하는 메커니즘을 도입하였다.
3. Hindsight notes를 포함한 노트 테이킹 시스템으로 세션 간 지속적 학습을 구현하였다.
4. Meta-agent를 통해 에이전트 설계 자체를 자동화하는 build-test-improve 루프를 제안하였다.
5. SWE-Bench-Pro에서 Resolve@1 59%를 달성하여, 동일 모델·동일 도구 접근 조건에서 기존 시스템을 초과하였다.
6. 약한 모델 + 강한 스캐폴딩(Claude 4.5 Sonnet + CCA: 52.7%)이 강한 모델 + 약한 스캐폴딩(Claude 4.5 Opus + Anthropic: 52.0%)을 능가함을 실증하였다.

## Experiment & Results
- **SWE-Bench-Pro (731 tasks)**: GPT-5.2 + CCA가 **59.0%** Resolve@1을 달성하여 OpenAI 공식 결과(56.0%)를 3%p 초과.
- Claude 4 Sonnet + CCA는 **45.5%**로, 동일 모델의 SWE-Agent(42.7%) 대비 +2.8%p 향상.
- Claude 4.5 Sonnet + CCA는 **52.7%**로, Live-SWE-Agent(45.8%) 대비 +6.9%p 향상.
- Claude 4.5 Opus + CCA는 **54.3%**로 Anthropic 공식 결과(52.0%)를 초과.
- **컨텍스트 관리 ablation (100개 subset)**: 컨텍스트 관리 적용 시 42.0% → **48.6%** (+6.6%p) 향상.
- **다중 파일 편집 견고성**: 1-2 파일 57.8%, 3-4 파일 49.2%, 5-6 파일 44.1%, 10+ 파일 44.4%.
- **노트 테이킹 효과 (151 instances)**: Run 2에서 평균 턴 수 64→**61**, 토큰 비용 104k→**93k** (-11k), Resolve Rate 53.0%→**54.4%** (+1.4%p).
- **SWE-Bench-Verified (500 tasks)**: Claude 4 Sonnet + CCA가 **74.6%**로 OpenHands(72.8%)를 초과.
- Thinking budget 스케일링: 8k(67.3%) → 16k(68.4%) → 32k(68.7%)로 16k 이후 수익 체감.
- 컨텍스트 요약 품질 ablation: Sonnet 4 요약기가 50개 인스턴스에서 **26/50**, Haiku가 **22/50** 해결.

---

## Limitation
- 다중 파일 편집(10+ 파일) 시나리오에서 44.4%로 성능 하락이 관찰되며, 다중 파일 의존성 추적이 부족하다.
- 노트 테이킹의 성능 향상 폭이 +1.4%p로 제한적이며, 장기적 노트 축적의 스케일링 효과 검증이 부족하다.
- Meta-agent의 build-test-improve 루프 수렴 시간과 비용에 대한 정량적 분석이 제시되지 않았다.
- Claude Code와의 직접 비교가 실행 환경 차이로 불가능하여, 공정한 상용 시스템 대비 평가가 제한적이다.
- 평가가 Python 리포지토리 중심(SWE-Bench)에 편중되어 있으며, 다른 프로그래밍 언어로의 일반화가 검증되지 않았다.
- Architect Agent의 컨텍스트 압축 품질이 요약 모델 능력에 의존하며(Haiku vs Sonnet 4%p 차이), 약한 모델 사용 시 성능 저하가 발생할 수 있다.
