# 논문 정리 전체 파이프라인

## 개요

논문 정리 요청이 들어오면 아래 파이프라인을 **순서대로** 수행한다.

```
[논문 정리 요청]
  → Step 1: 논문 PDF 확보 및 저장
  → Step 2: 요약(Summary) 마크다운 작성
  → Step 3: 개요 파일(agent_memory_papers.md) 업데이트
  → Step 4: Notion 업로드
```

---

## Step 1: 논문 PDF 확보 및 저장

1. 사용자가 논문 제목, arXiv ID, 또는 PDF 파일을 제공
2. PDF가 없으면 arXiv 등에서 다운로드
3. `agent_memory_pdfs/` 폴더에 저장
   - 파일명 규칙: `{논문약칭}.pdf` (예: `A-MEM.pdf`, `MAGMA.pdf`)
4. PDF가 텍스트 추출이 어려운 경우 `{논문약칭}_text.txt`로 텍스트 버전도 저장

## Step 2: 요약(Summary) 마크다운 작성

### 파일 위치 및 명명 규칙

- 경로: `summaries/{번호}_{논문약칭}.md`
- 번호: `agent_memory_papers.md`의 기존 논문 수 + 1 (2자리, 01~99)
- 예: `summaries/14_NewPaper.md`

### 요약 템플릿

아래 구조를 따라 작성한다. PDF를 읽고 각 항목을 채운다.

**핵심 원칙**: Problem/Motivation/Limitation은 테이블로 간결하게, Method/Key Contribution/Experiment는 독립 섹션으로 분리하여 충분히 상세하게 작성한다. 논문을 읽지 않은 사람이 이 요약만으로 핵심 메커니즘과 실험 결과를 파악할 수 있어야 한다.

```markdown
# {논문 정식 제목}

> **논문 정보**: {저자} ({소속})
> **arXiv**: {arXiv ID} ({날짜})
> **코드**: {GitHub URL 있으면 기재}

---

## Overview

| 항목 | 내용 |
|------|------|
| **Problem** | {이 논문이 해결하려는 문제. 기존에 왜 안 됐는가 — 2~4줄} |
| **Motivation** | {왜 이 문제가 중요한가. 실제 한계/실패 사례 — 2~4줄} |
| **Limitation** | {저자가 밝힌 한계 + 읽으면서 느낀 한계 — 3~5줄} |

---

## Method

{논문의 핵심 방법론을 구조적으로 서술한다. 아래 가이드라인을 따른다.}

- **분량**: 15~25줄 (논문 복잡도에 따라 조정)
- **구조**: 번호 매긴 단계별 설명, 각 단계 내에서 하위 bullet 활용
- **포함할 내용**:
  - 전체 아키텍처/파이프라인의 흐름 (입력 → 처리 → 출력)
  - 각 핵심 컴포넌트의 역할과 동작 방식
  - 핵심 알고리즘/수식이 있으면 직관적으로 설명 (수식 자체보다 의미 중심)
  - 기존 방법과 구조적으로 어떻게 다른지
  - 서베이 논문의 경우: 분류 체계(taxonomy)의 축과 각 범주 설명

---

## Key Contribution

{기존 연구 대비 novelty를 번호 매겨 나열한다.}

- **분량**: 5~10줄
- 각 항목은 "무엇을" + "왜 중요한지"를 포함
- 단순 나열이 아닌, 각 기여가 어떤 기존 한계를 해결하는지 연결

---

## Experiment & Results

{벤치마크, 실험 설계, 주요 수치를 구체적으로 기술한다.}

- **분량**: 10~20줄 (실험 논문), 5~10줄 (서베이/이론 논문)
- **포함할 내용**:
  - 사용한 데이터셋/벤치마크 (규모, 특성)
  - 비교 대상 (baseline 모델들)
  - 핵심 정량 결과 — 구체적 수치 필수 (예: "F1=45.85 vs MemGPT 25.52")
  - Ablation study 결과 (있는 경우)
  - 비용/효율성 수치 (있는 경우)
  - 정성적 분석 또는 case study 요약 (의미 있는 경우)

---

## 선택 요소 (해당되는 것만)

| 항목 | 내용 |
|------|------|
| **Taxonomy/분류 체계** | {서베이 논문의 경우 분류 축} |
| **Threat Model** | {보안 논문의 경우 공격자 모델} |
```

### 작성 가이드라인

1. **Problem → Method → Result 흐름**을 유지하면 논문 간 비교가 쉬워진다
2. **Method는 이 요약의 핵심이다**. 논문을 안 읽은 사람이 메커니즘을 이해할 수 있도록 충분히 구조적으로 작성한다. 단순히 "X를 제안함"이 아니라 "X가 어떻게 동작하는지"를 설명한다
3. **Experiment는 구체적 수치를 반드시 포함**한다. "성능이 좋다" 대신 "F1=45.85 vs MemGPT 25.52 (+80%)"
4. **Limitation은 저자 언급 + 독자 관점** 두 가지 모두 기록한다
5. **Baseline 비교와 Ablation은 Experiment 섹션에 통합**한다. 별도 선택 요소로 분리하지 않는다
6. **서베이 논문**의 경우: Method 대신 분류 체계를, Experiment 대신 커버리지 분석을 중심으로 작성한다

## Step 3: 개요 파일 업데이트

`agent_memory_papers.md`에 새 논문을 추가한다.

1. 논문이 속하는 카테고리를 판단 (기존 카테고리 중 택1, 또는 새 카테고리 생성)
2. 해당 카테고리 테이블에 새 행 추가 (5열 구조):
   ```
   | **{논문약칭}** | {발표 날짜} | {학회/발표기관} | {1저자} | {핵심 아이디어 1줄 요약} |
   ```

## Step 4: Notion 업로드

`notion_mcp_instruction.md`의 파이프라인을 따라 Notion에 반영한다.

1. **(필수) 기존 블록 구조 조회**: `GET /blocks/{메인페이지ID}/children`로 현재 페이지 구조 파악
2. **메인 페이지 개요 테이블 업데이트**:
   - 기존 카테고리에 추가 → 테이블 삭제 후 재생성
   - 새 카테고리 → 마지막 카테고리 테이블 뒤에 `after` 파라미터로 삽입 (단순 append 금지)
3. **하위 논문 상세 페이지 생성** (`POST /pages`, 토글 블록 형태)
4. **Callout 업데이트**: 논문 수/카테고리 수 반영
5. **검증**: 블록 순서가 `[카테고리들] → [Divider] → [하위 페이지들]` 구조인지 확인

상세 Notion API 호출 방법은 `notion_mcp_instruction.md` 참조.

```
paper_listup/
├── CLAUDE.md                    # Claude 에이전트 지침서
├── summarize_instruction.md     # 이 파일 (전체 파이프라인)
├── notion_mcp_instruction.md    # Notion 업로드 상세 가이드
├── agent_memory_papers.md       # 전체 논문 개요 (카테고리별 테이블)
├── agent_memory_pdfs/           # 원본 PDF 저장소
│   ├── A-MEM.pdf
│   ├── MAGMA.pdf
│   └── ...
└── summaries/                   # 개별 논문 요약 마크다운
    ├── 01_A-MEM.md
    ├── 02_MAGMA.md
    └── ...
```
