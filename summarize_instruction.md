# 논문 정리 전체 파이프라인

## 개요

논문 정리 요청이 들어오면 아래 파이프라인을 **순서대로** 수행한다.

```
[논문 정리 요청]
  → Step 1: 논문 PDF 확보 및 저장
  → Step 2: 요약(Summary) 마크다운 작성 (서브에이전트로 PDF 분석 → 메인에서 파일 저장)
  → Step 2.5: 품질 검증 (분량·수치 체크, 미달 시 PDF 재독 후 보강)
  → Step 3: 개요 파일(agent_memory_papers.md) 업데이트
  → Step 4: Notion 업로드 (마크다운 파일을 읽어서 반영)
```

---

## Step 1: 논문 PDF 확보 및 저장

1. 사용자가 논문 제목, arXiv ID, 또는 PDF 파일을 제공
2. PDF가 없으면 arXiv 등에서 다운로드
3. `agent_memory_pdfs/` 폴더에 저장
   - 파일명 규칙: `{논문약칭}.pdf` (예: `A-MEM.pdf`, `MAGMA.pdf`)
4. PDF가 텍스트 추출이 어려운 경우 `{논문약칭}_text.txt`로 텍스트 버전도 저장

## Step 2: 요약(Summary) 마크다운 작성

### 파일 위치 및 명명 규칙

- 경로: `summaries/{번호}_{논문약칭}.md`
- 번호: `agent_memory_papers.md`의 기존 논문 수 + 1 (2자리, 01~99)
- 예: `summaries/14_NewPaper.md`

### 요약 템플릿

아래 구조를 따라 작성한다. PDF를 읽고 각 항목을 채운다.

**핵심 원칙**: 6개 섹션(Problem, Motivation, Method, Key Contribution, Experiment, Limitation) 모두 독립 섹션으로 충분히 상세하게 작성한다. 논문을 읽지 않은 사람이 이 요약만으로 핵심 메커니즘과 실험 결과를 파악할 수 있어야 한다. **모든 섹션의 내용을 합치면 최소 A4 한 페이지 분량(80줄 이상)이 되어야 한다.**

```markdown
# {논문 정식 제목}

> **논문 정보**: {저자} ({소속})
> **arXiv**: {arXiv ID} ({날짜})
> **코드**: {GitHub URL 있으면 기재}

---

## Problem

{이 논문이 해결하려는 문제를 구체적으로 서술한다.}

- **분량**: 5~8줄
- **포함할 내용**:
  - 기존 접근법의 구체적 한계 (어떤 상황에서, 왜 실패하는지)
  - 문제의 기술적 원인 (단순 "어렵다"가 아닌 구체적 병목)
  - 관련 선행 연구가 이 문제를 어떻게 다뤘고 왜 부족했는지
- 예시: "기존 RAG 시스템은 단일 검색으로 답을 찾으려 하지만, 다단계 추론이 필요한 질문에서는 첫 검색 결과가 불충분하여 평균 정확도가 40% 이하로 떨어진다. ReAct 기반 접근은 반복 검색을 시도하지만, 검색 쿼리 생성이 비효율적이고 불필요한 반복으로 지연이 발생한다."

---

## Motivation

{왜 이 접근법을 선택했는지, 핵심 직관(insight)을 설명한다.}

- **분량**: 5~8줄
- **포함할 내용**:
  - 핵심 관찰/직관 (이 논문의 "아하!" 모먼트)
  - 실제 한계/실패 사례가 있으면 구체적으로 기술
  - 인간의 인지 과정이나 다른 분야에서의 유사한 해결 패턴 (해당 시)
  - 이 접근이 기존과 근본적으로 어떻게 다른 관점인지
- 예시: "인간은 복잡한 질문에 답할 때 한번에 정보를 찾지 않고, 부분 답을 얻은 후 다음 질문을 정교화한다. 이를 LLM 에이전트에 적용하면, 각 검색 단계에서 이전 결과를 반영하여 쿼리를 재구성함으로써 다단계 추론의 정확도를 크게 높일 수 있다."

---

## Method

{논문의 핵심 방법론을 구조적으로 서술한다.}

- **분량**: 15~25줄 (논문 복잡도에 따라 조정)
- **구조**: 번호 매긴 단계별 설명, 각 단계 내에서 하위 bullet 활용
- **포함할 내용**:
  - 전체 아키텍처/파이프라인의 흐름 (입력 → 처리 → 출력)
  - 각 핵심 컴포넌트의 역할과 동작 방식
  - 핵심 알고리즘/수식이 있으면 직관적으로 설명 (수식 자체보다 의미 중심)
  - 기존 방법과 구조적으로 어떻게 다른지
  - 서베이 논문의 경우: 분류 체계(taxonomy)의 축과 각 범주 설명

---

## Key Contribution

{기존 연구 대비 novelty를 번호 매겨 나열한다.}

- **분량**: 5~10줄
- 각 항목은 "무엇을" + "왜 중요한지"를 포함
- 단순 나열이 아닌, 각 기여가 어떤 기존 한계를 해결하는지 연결

---

## Experiment & Results

{벤치마크, 실험 설계, 주요 수치를 구체적으로 기술한다.}

- **분량**: 10~20줄 (실험 논문), 5~10줄 (서베이/이론 논문)
- **포함할 내용**:
  - 사용한 데이터셋/벤치마크 (규모, 특성)
  - 비교 대상 (baseline 모델들)
  - 핵심 정량 결과 — 구체적 수치 필수 (예: "F1=45.85 vs MemGPT 25.52")
  - Ablation study 결과 (있는 경우)
  - 비용/효율성 수치 (있는 경우)
  - 정성적 분석 또는 case study 요약 (의미 있는 경우)

---

## Limitation

{이 논문의 한계를 구체적으로 분석한다.}

- **분량**: 5~8줄
- **포함할 내용**:
  - 저자가 명시적으로 밝힌 한계
  - 읽으면서 느낀 추가 한계 (평가 범위, 확장성, 가정의 현실성 등)
  - 각 한계가 실제 적용 시 어떤 영향을 미치는지
  - 가능한 해결 방향 (간략히)

---

## 선택 요소 (해당되는 것만)

| 항목 | 내용 |
|------|------|
| **Taxonomy/분류 체계** | {서베이 논문의 경우 분류 축} |
| **Threat Model** | {보안 논문의 경우 공격자 모델} |
```

### 작성 가이드라인

1. **Problem → Motivation → Method → Result → Limitation 흐름**을 유지하면 논문 간 비교가 쉬워진다
2. **모든 6개 섹션이 독립적으로 충실해야 한다**. 어떤 섹션도 1-2줄로 끝나서는 안 된다
3. **Problem은 구체적 실패 원인**을 서술한다. "기존 방법은 한계가 있다" 대신 "기존 방법 X는 Y 상황에서 Z 이유로 실패한다"
4. **Motivation은 핵심 직관(insight)**을 전달한다. 이 논문의 접근이 왜 합리적인지, 어떤 관찰에서 출발했는지
5. **Method는 이 요약의 핵심이다**. 논문을 안 읽은 사람이 메커니즘을 이해할 수 있도록 충분히 구조적으로 작성한다. 단순히 "X를 제안함"이 아니라 "X가 어떻게 동작하는지"를 설명한다
6. **Experiment는 구체적 수치를 반드시 포함**한다. "성능이 좋다" 대신 "F1=45.85 vs MemGPT 25.52 (+80%)"
7. **Limitation은 저자 언급 + 독자 관점** 두 가지 모두 기록하고, 실제 적용 시 영향까지 분석한다
8. **Baseline 비교와 Ablation은 Experiment 섹션에 통합**한다. 별도 선택 요소로 분리하지 않는다
9. **서베이 논문**의 경우: Method 대신 분류 체계를, Experiment 대신 커버리지 분석을 중심으로 작성한다

## Step 2.5: 품질 평가 루프 (필수 — 건너뛰지 않는다)

각 논문 요약에 대해 **독립된 평가 에이전트**가 품질을 검증한다.
**논문 1편당 1개 평가 에이전트**를 실행하여 컨텍스트 오버로드를 방지한다.

### 평가 루프 흐름

```
[배치 요약 완료 (3편)]
  → 논문 1편당 1개 평가 에이전트 실행 (3개 병렬)
    → 각 에이전트: 요약 1편 + 원본 PDF 텍스트 1편만 읽고 평가
    → PASS / FAIL + 구체적 피드백 반환
  → FAIL 논문에 대해 1편당 1개 보강 에이전트 실행 (병렬)
    → 원본 PDF를 다시 읽고 부족한 부분을 보강하여 반환
  → 메인에서 Write로 보강 내용 반영
  → 재평가 (최대 2회 반복, 역시 1편당 1개 에이전트)
  → 전수 PASS 시 Step 3으로 진행
```

### 컨텍스트 관리 원칙

1. **1 에이전트 = 1 논문**: 평가·보강 에이전트는 반드시 논문 1편만 담당. 여러 논문을 한 에이전트에 넣지 않는다
2. **병렬 실행**: 배치 내 논문들의 평가/보강 에이전트를 동시에 실행하여 속도 확보
3. **메인 컨텍스트 최소화**: 서브에이전트 반환값을 Write로 저장한 직후, 다음 논문으로 즉시 진행. 이전 논문의 상세 내용을 메인에서 재참조하지 않는다
4. **배치 크기 3편**: 한 배치에 최대 3편만 처리. 3편 완료(요약+평가+보강) 후 다음 배치로 넘어간다

### 평가 에이전트 실행 방법

**논문 1편당 1개 에이전트**를 실행한다. 배치 내 3편이면 3개 에이전트를 **병렬로** 실행:

```
Agent 도구 (subagent_type="general-purpose") 프롬프트:

"당신은 논문 요약의 품질 평가자입니다. 아래 1편의 요약을 원본 PDF 텍스트와 비교하여 평가하세요.

평가 대상:
- 요약: summaries/{번호}_{약칭}.md
- 원본: agent_memory_pdfs/{약칭}_text.txt

평가 기준:
[A] 분량 (정량)
  - 전체: 80줄 이상
  - Problem 섹션: 5줄 이상, 기존 접근법의 구체적 한계 포함
  - Motivation 섹션: 5줄 이상, 핵심 직관/관찰 포함
  - Method 섹션: 15줄 이상, 번호 매긴 단계별 구조
  - Experiment 섹션: 10줄 이상
  - Limitation 섹션: 5줄 이상, 저자 언급 + 독자 관점
  - 구체적 수치 (%, 소수점, +N, vs): 5개 이상

[B] 충실도 (정성) — 원본 PDF와 비교
  - Problem이 '어렵다' 수준이 아닌, 기존 방법의 구체적 실패 원인을 서술하는가?
  - Motivation이 핵심 직관/관찰을 포함하며, 왜 이 접근이 합리적인지 설명하는가?
  - Method가 '~를 제안함' 수준이 아닌, 구체적 동작 방식을 서술하는가?
  - Experiment에 PDF의 주요 벤치마크/수치가 빠짐없이 포함되었는가?
  - Limitation에 저자 언급 + 독자 관점이 모두 있으며, 실제 영향이 분석되었는가?

반환 형식:
---
## {번호}_{약칭}
- 판정: PASS / FAIL
- 전체 줄수: {N}줄
- Problem 줄수: {N}줄
- Motivation 줄수: {N}줄
- Method 줄수: {N}줄
- Experiment 줄수: {N}줄
- Limitation 줄수: {N}줄
- 수치 개수: {N}개
- [A] 분량 판정: PASS / FAIL (사유)
- [B] 충실도 판정: PASS / FAIL (사유)
- 보강 필요 사항: (FAIL인 경우 구체적으로 무엇을 추가해야 하는지)
---

파일을 직접 수정하지 말고, 평가 결과만 텍스트로 반환하세요."
```

### FAIL 논문 보강

평가에서 FAIL이 나온 논문에 대해 **1편당 1개 보강 에이전트**를 실행한다 (여러 FAIL이면 병렬):

```
Agent 도구 (subagent_type="general-purpose") 프롬프트:

"아래 논문 요약을 보강하세요.

- 현재 요약: summaries/{번호}_{약칭}.md (Read로 읽을 것)
- 원본 PDF: agent_memory_pdfs/{약칭}_text.txt (Read로 읽을 것)
- 평가 피드백: {평가 에이전트가 반환한 해당 논문의 피드백}

보강 지침:
- 평가에서 지적된 부족한 부분만 보강 (잘 작성된 부분은 유지)
- Problem: 5~8줄, 기존 접근법의 구체적 실패 원인
- Motivation: 5~8줄, 핵심 직관/관찰, 왜 이 접근이 합리적인지
- Method: 15~25줄, 번호 매긴 단계별, 각 컴포넌트 동작 방식
- Experiment: 10~20줄, 구체적 수치 필수 (PDF 테이블에서 추출)
- Limitation: 5~8줄, 저자 언급 + 독자 관점 + 실제 영향
- 빠진 수치/벤치마크가 있으면 PDF에서 찾아서 추가

보강된 전체 마크다운을 텍스트로 반환하세요. 파일 쓰기는 하지 마세요."
```

메인 대화에서 반환된 보강 내용을 Write로 저장한 후, 필요 시 재평가한다 (최대 2회, 역시 1편당 1개 에이전트).

---

## Step 3: 개요 파일 업데이트

`agent_memory_papers.md`에 새 논문을 추가한다.

1. 논문이 속하는 카테고리를 판단 — 아래 6개 중 택1:
   - **1. Memory Management** — 메모리 저장/검색/구조화/진화
   - **2. Tool Use & MCP** — 외부 도구 호출, function calling, MCP
   - **3. Planning & Task Decomposition** — 작업 분할, 계획 수립, 오케스트레이션
   - **4. Self-Correction & Test-Time Compute** — 자가 교정, 추론 시간 연산, RL 기반 학습
   - **5. Agentic RAG & Graph RAG** — 지식 그래프, 그래프 RAG, 능동적 검색
   - **6. General** — 위 5개에 해당하지 않는 에이전트 아키텍처/서베이/기타
2. 해당 카테고리 테이블에 새 행 추가 (5열 구조):
   ```
   | **{논문약칭}** | {발표 날짜} | {학회/발표기관} | {1저자} | {핵심 아이디어 1줄 요약} |
   ```

## Step 4: Notion 업로드

`notion_mcp_instruction.md`의 파이프라인을 따라 Notion에 반영한다.

1. **(필수) 기존 블록 구조 조회**: `GET /blocks/{메인페이지ID}/children`로 현재 페이지 구조 파악
2. **메인 페이지 개요 테이블 업데이트**:
   - 기존 카테고리에 추가 → 테이블 삭제 후 재생성
   - 새 카테고리 → 마지막 카테고리 테이블 뒤에 `after` 파라미터로 삽입 (단순 append 금지)
3. **하위 논문 상세 페이지 생성** (`POST /pages`, 토글 블록 형태)
   - **⚠ 반드시 로컬 마크다운 파일(`summaries/{번호}_{약칭}.md`)을 Read로 읽어서** 내용을 Notion에 반영할 것
   - 별도로 내용을 재작성하거나 하드코딩하지 않는다 — 마크다운의 각 섹션을 파싱하여 Notion 블록으로 변환
4. **Callout 업데이트**: 논문 수/카테고리 수 반영
5. **검증**: 블록 순서가 `[카테고리들] → [Divider] → [하위 페이지들]` 구조인지 확인

상세 Notion API 호출 방법은 `notion_mcp_instruction.md` 참조.
